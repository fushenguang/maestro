# 如何设计真实案例并稳定通过 Validation（含 MagicMirror² 实战）

## 1. 这份文档解决什么问题

这份指南用于避免在 Maestro 的 Phase 3（Validation）卡住，尤其是出现 `NO-GO` 或 `fatal_risk` 后流程无法推进。

目标是让利益相关者与 AI Agent 都能用同一套方法，设计出：

1. 真实可卖（有明确付费用户与价格区间）
2. 真实可做（12 周内可交付 MVP）
3. 真实可验证（Validation 通过概率高）
4. 真实可演化（可继续进入 Contract 与 Evolution）

---

## 2. 先理解代码里的“硬判定逻辑”

当前 Validation 结论并非 UI 文案，而是硬逻辑计算。

关键规则（来自前端逻辑）：

1. 任意证据项出现 `fatal_risk` => 直接 `no_go`
2. 评分 `score <= -2` => `no_go`
3. 检方（prosecutor）中 `adoption_risk + evidence_gap` 数量 `>= 4` => `no_go`
4. `score >= 3` 且检方风险数 `<= 2` => `go`
5. 其他情况 => `pending`

设计案例时必须“反向工程”这个规则，而不是只写愿景。

---

## 3. 为什么会被判 NO-GO（复盘模板）

典型失败模式：

1. 用户价值与实施内容错位
   例：目标用户是硬件玩家，但方案强调的是“内部架构重构”。
2. 付费主体不成立
   例：把 AI Agent 当“用户”，但 AI Agent 不会付费也不做采购决策。
3. 关键交付链路断裂
   例：核心价值依赖 C/C++ 控制，却把 C/C++ 互操作排除在外。
4. Scope 过大或含混
   例：把“硬件量产 + 内容平台 + App + B2B SaaS”同时塞进 MVP。

## 4. 设计真实案例的 8 步法

### Step 1: 定义“谁付钱”

必须明确：

1. 购买人群（谁掏钱）
2. 使用人群（谁每天用）
3. 传播人群（谁会帮你带来流量）

三者可以重叠，但不能缺失。

### Step 2: 定义“单点可感知价值”

MVP 只能有一个主价值：

1. 节省时间
2. 提升收入
3. 降低焦虑
4. 提升形象/社交货币

### Step 3: 用数字写出商业闭环

至少给出：

1. 成本结构（BOM）
2. 售价区间
3. 毛利区间
4. 交付周期
5. 售后边界

### Step 4: 明确不可做（Out of Scope）

Validation 阶段最怕“什么都做”。

必须写清：

1. 这期不做什么
2. 为什么不做
3. 不做会不会影响核心价值（如果会，说明不能列为 out）

### Step 5: 预先拆解 fatal risk

把可能被 prosecutor 判成 `fatal_risk` 的点写成“已知风险 + 缓解策略”。

### Step 6: 证据类型配比（用于通过 Validation）

建议目标：

1. Advocate: 4-6 条，`proves_problem` 占主导
2. Prosecutor: 尽量避免 `fatal_risk`，`adoption_risk + evidence_gap` 控制在 2 条以内

### Step 7: 写成“可执行叙述”

不要写“打造生态、重塑行业”。

要写：

1. 用户今天怎么做
2. 你的方案替代了哪 3 个步骤
3. 用户 7 天后会看到什么可量化变化

### Step 8: 预先准备 Phase 4/5 素材

在进入 Contract 前先想好：

1. deadline
2. success_metric
3. target_n
4. github_repo

否则会在 Contract 阶段继续卡住。

## 5. 实战案例：AI 语音智能魔镜（MagicMirror² + DeepSeek + P1S 边框）

## 5.1 案例定位（一句话）

面向租房青年与轻度智能家居用户的“早晚出门决策屏”，把时间、天气、日历、提醒、新闻和语音交互整合到一面可定制智能镜子中，兼具实用价值与传播价值。

## 5.2 商业价值与价格带

1. 成本目标（MVP BOM）：
   1. 树莓派（或同级板卡）：¥260-350
   2. 单向镜膜与辅材：¥60-120
   3. 旧显示器翻新成本：¥0-200（回收渠道）
   4. 麦克风与小音箱：¥80-150
   5. P1S 打印边框材料与工时：¥80-150
   合计：约 ¥480-800
2. 建议售价：¥1500-2500
3. 单台毛利空间：约 ¥700-1700
4. 商业增益：
   1. 硬件销售毛利
   2. 可选安装服务费
   3. 内容素材天然适合短视频传播，降低获客成本

## 5.3 用户与场景

1. 购买人群：愿意为“科技感 + 实用性”付费的租房青年、小家庭
2. 使用人群：早晚通勤者（出门前 3 分钟）
3. 核心场景：
   1. 早晨：看天气/日程/提醒，语音问“今天要带伞吗？”
   2. 晚间：查看明日安排，语音快速创建提醒

## 5.4 MVP 功能边界（给 Boundary 阶段直接用）

### In Scope（必须做）

1. 镜面实时卡片：时间、天气、今日日历事件、头条摘要
2. 语音唤醒 + 意图识别（DeepSeek）
3. 语音指令 3 类：天气问答、日程查询、提醒创建
4. 可替换主题（2 套）与亮度时段策略
5. P1S 打印可选边框 2 款（极简黑、木纹风）

### Out of Scope（必须明确不做）

1. 全屋中控协议统一（HomeKit/Matter 全量适配）
2. 视频通话与摄像头识别
3. 多语言同传
4. 自建语音大模型训练
5. 批量量产供应链系统

### Open Questions（最多 1-2 个）

1. 首发是否只支持国内天气源（减少 API 不稳定）
2. 提醒同步首发仅支持 Google Calendar，还是本地离线提醒也上

## 6. 直接可用的 Phase 输入素材

## 6.1 Feed 原始输入（可直接粘贴）

我们做一款 AI 语音智能魔镜，核心是把 MagicMirror² 和树莓派组合成可售卖的家用硬件产品。用户在镜子前可以直接看到时间、天气、日历和新闻，并通过语音（接 DeepSeek）进行查询和创建提醒。硬件通过单向镜膜 + 旧显示器 + P1S 打印边框实现，成本控制在 800 元以内，售价 1500-2500 元。这个产品有明确用户价值（出门决策和效率提升），也有天然传播素材价值（黑科技视觉效果），可以做小规模工作室交付。

## 6.2 Intent 目标答案（用于把 clarity 提到 85+）

1. problem:
   家庭与租房用户每天出门前需要在多个设备间反复查看天气、日程和提醒，信息分散导致决策慢、遗漏多。
2. root_cause:
   关键决策信息没有在“起床-洗漱-出门”动线上集中展示，且语音交互入口零散。
3. mechanism:
   用镜面显示卡片聚合核心信息，再用 DeepSeek 语音意图解析完成查询与提醒创建，形成 3 分钟出门决策闭环。
4. target_user:
   一二线城市租房青年、注重效率与科技感的小家庭用户。
5. success_metric_desc:
   30 天内完成首批 50 台交付，用户 7 日留存中“每日至少一次语音交互”达到 40%。

提示：Intent 阶段避免抽象词（如“生态”“平台化”），用“谁-在何时-遇到什么痛点-你怎么替代”表达。

## 6.3 Boundary 锁定前检查

1. In Scope 是否都能在 12 周内做完
2. Out of Scope 是否覆盖所有“看起来很酷但会拖死项目”的方向
3. Open Questions 是否 <= 2，且有明确决策人

---

## 7. Validation 通关策略（关键）

## 7.1 让 Advocate 提供强证据的素材

建议在想法描述里提前放入这些可证据化点：

1. 智能家居可视化需求增长
2. 旧显示器改造与 DIY 社群成熟
3. 短视频平台对“黑科技家居”内容偏好明显
4. 成本和售价形成可观毛利区间

## 7.2 降低 Prosecutor 触发 fatal_risk 的方法

1. 绝不把“内部工程优化”写成主要价值
2. 明确“付费用户是谁”，避免“AI Agent 是用户”这类 category error
3. 不回避关键依赖，写清已有可行替代方案
4. 风险要写“可缓解动作”，而不是“后续再看”

## 7.3 目标风险结构（建议）

1. prosecutor 的 `adoption_risk + evidence_gap` 最好 <= 2
2. 避免出现 `fatal_risk`
3. 即使有风险，也要让 advocate 的 `proves_problem` 数量与强度足够高

## 8. 手动验证脚本（完整流程）

这个脚本用于完成一次从 Validation 到 Contract 再到 Evolution 的真实走查。

## 8.1 准备

1. 启动桌面应用
2. 创建 idea，粘贴 6.1 的 Feed 文本
3. 在 Intent 回答使用 6.2 的五项结构
4. 在 Boundary 采用 5.4 的 in/out/open 列表并锁定

## 8.2 Validation 阶段验收

1. 进入 Validation，等待 Advocate 与 Prosecutor 两轮结束
2. 验收标准：
   1. 不出现 `fatal_risk`
   2. 若出现风险，`adoption_risk + evidence_gap` 不超过 2
   3. verdict 为 `go` 或至少 `pending`（若 pending 可先补证据后 rerun）
3. 点击 `Accept & proceed` 后应进入 Contract 页面

## 8.3 Contract 阶段验收（建议参数）

1. product_type: `opensource`
2. deadline: 当前日期 + 90 天
3. success_metric: `github_stars`
4. target_n: `1000`
5. github_repo: `owner/repo`
6. 勾选 checklist，执行两步确认签署
7. 验收标准：
   1. 字段进入只读
   2. 显示 contract_ref 与签署时间
   3. 若导出失败，出现可重试提示

## 8.4 Evolution 阶段验收（建议节点）

1. 新建 milestone:
   1. `v0.1.0` 核心看板 + 语音查询
   2. `v0.2.0` 语音提醒创建 + 边框 SKU A/B
2. 验收标准：
   1. 节点按顺序展示
   2. scope warning 节点会阻断 trigger openspec
   3. 输入 dismiss reason 后可解锁
   4. openspec change 需先写 arch log 才能 done

## 9. 给 AI Agent 的执行提示

当 Agent 生成案例时，必须满足：

1. 用户价值为主，不得以内部重构为主价值
2. 明确付费主体、交付对象、传播对象
3. 输出必须包含：
   1. problem
   2. mechanism
   3. target_user
   4. success_metric
   5. in_scope/out_of_scope/open_question
   6. 成本与售价区间
4. 若发现“可能触发 fatal_risk”，先改案例再进入 Validation

## 11. NO-GO 后的重构模板（针对本次失败）

本次检方给出的 fatal risk 有 4 个：

1. 硬件可复制，难以支撑高售价
2. 浴室语音场景文化与租房限制不成立
3. 声学与延迟在浴室环境不可靠
4. DeepSeek 作为 voice-first 核心不够可信

重构原则：

1. 从“卖硬件差价”改为“卖交付结果（安装 + 配置 + 模板内容 + 售后）”
2. 从“浴室镜前语音”改为“玄关/梳妆台/书桌的低噪场景，语音是增强而不是唯一入口”
3. 从“DeepSeek 全链路”改为“ASR/TTS 用成熟组件，DeepSeek 只做意图理解与摘要生成”
4. 明确“租房友好”约束：不打孔、可搬走、10 分钟安装

### 11.1 重构后的一句话定位

这是一个“租住友好晨间决策镜屏服务”，通过免打孔安装的镜屏终端 + 预配置信息模板 + 可选语音交互，帮助通勤用户在 3 分钟内完成出门决策；我们卖的是开箱即用交付与持续优化，而不是单纯硬件。

### 11.2 重构后 Feed 输入（可直接粘贴）

我们不是卖一面可复制的智能镜硬件，而是卖“租住友好的晨间决策镜屏交付服务”。产品放在玄关、梳妆台或书桌场景（不限定浴室），10 分钟免打孔安装、可搬家带走。屏上固定显示时间、天气、日程、提醒和本地化资讯，语音是增强入口，不可用时可以触控/旋钮完成同样操作。技术上采用成熟 ASR/TTS 方案保障实时性，DeepSeek 仅用于意图理解与信息摘要。单台硬件成本控制在 800 元内，售价分为硬件包（1499）与交付服务包（1999-2499，含安装配置与 30 天模板迭代），首批目标在本地城市完成 50 台付费交付。

### 11.3 重构后 Intent 五要素（可直接使用）

1. problem:
   通勤用户出门前要在多个 App 来回切换确认天气、日程、提醒，容易遗漏，且手机信息流会打断决策。
2. root_cause:
   关键信息没有被放进“出门前固定动线”，并且现有智能家居方案对租房用户安装门槛高、迁移成本高。
3. mechanism:
   通过免打孔镜屏终端把决策信息聚合到固定位置，默认可视卡片即用，语音仅做加速入口；ASR/TTS 负责实时交互，DeepSeek 负责意图理解与摘要。
4. target_user:
   一二线城市 23-35 岁租房通勤人群，重视效率与空间美观，接受 1500-2500 元一次性投入换取每日高频使用价值。
5. success_metric_desc:
   首批 50 台付费用户中，14 天内“每周至少 5 天完成晨间 3 分钟决策流程”的用户占比 >= 60%，且 30 天退款率 <= 8%。

### 11.4 重构后 Boundary 建议

In Scope:

1. 玄关/桌面镜屏卡片（时间、天气、日程、提醒、资讯）
2. 免打孔安装方案与搬家迁移方案
3. 语音 + 触控双通道（语音失败可回退）
4. 城市级模板包（通勤提醒、天气预警、日程摘要）
5. 交付 SOP（安装、配置、7 日回访）

Out of Scope:

1. 浴室潮湿强噪声环境下的语音刚性承诺
2. 自研 ASR/TTS 模型
3. 全屋中控联动
4. 大规模量产与全国铺货

Open Questions:

1. 首发城市是否限定为 1 个（便于交付密度与售后）
2. 服务包是否默认包含一次上门迁移

### 11.5 定价重构（回答“为什么不是硬件溢价”）

建议价格结构：

1. 标准硬件包：1499（只含设备）
2. 交付服务包：1999-2499（含安装、配置、30 天模板优化、一次远程复诊）

这样回答的好处：

1. 把价值锚点从“硬件本体”转到“持续可用结果”
2. 降低“被复制硬件”带来的致命攻击
3. 与租房用户真实痛点（安装/迁移/维护）直接绑定

### 11.6 针对本次 fatal risk 的标准回复模板

1. 关于“硬件可复制”：
   我们承认硬件可复制，因此商业模型不依赖硬件差价，而依赖交付效率、模板运营和售后复购。
2. 关于“浴室语音不成立”：
   我们已取消浴室作为主场景，首发定位玄关/桌面低噪场景，语音为增强而非唯一交互。
3. 关于“延迟与声学风险”：
   我们将语音链路拆分并设定验收阈值（P50 < 1.5s，P90 < 2.5s），同时提供触控回退，避免单点失败。
4. 关于“DeepSeek 不适合 voice-first”：
   DeepSeek 只承担 NLU 与摘要，不承担 ASR/TTS，语音底层使用成熟语音引擎。
