# 功能可用性与完整性测试案例（不改 Validation 机制）

## 1. 目标与原则

本案例用于验证 Maestro 全流程功能是否可用、是否完整：

1. Feed -> Intent Dialogue -> Boundary Definition -> Validation Gate -> Contract -> Evolution
2. 不修改 Validation 机制，不绕过风险评估
3. 优先选择低物理交付摩擦、低硬件依赖、价值链清晰的案例

本案例不是为了“强行通过”，而是为了在保留严格评估的前提下，验证系统功能链路是否稳定。

---

## 2. 新案例（推荐，v2）

案例名称：会后行动项质检器（Action QA）

一句话：
把中文会议纪要中的行动项做“完整性质检”，自动标记缺失字段（负责人/截止时间/优先级）并生成修复清单，帮助会议主持人在会后 10 分钟内补齐任务定义。

为什么选它：

1. 单用户即可闭环：不依赖团队成员系统，不触发协作架构 fatal risk
2. 价值更聚焦：先做“质检+补齐”，不承诺全流程任务管理
3. 与现有工具互补：不是替代聊天/PM，而是会后质量闸门
4. 可量化：行动项完整率、会后补齐时长、遗漏率下降

---

## 3. 可直接粘贴的 Feed 输入

我们做一个“会后行动项质检器”，面向 5-30 人团队的会议主持人或项目负责人。会议结束后，用户粘贴中文会议纪要，系统自动识别行动项并检查每条是否具备负责人、截止时间、优先级三个关键字段；对缺失字段给出修复建议和一键补齐清单。目标是减少会后任务定义不完整导致的执行遗漏。产品首版只做文本导入、行动项识别、缺失字段检测、修复清单导出，不做团队协同、推送中心或复杂项目管理。

---

## 4. Intent 阶段标准答案（用于快速收敛）

### 4.1 目标用户是谁？

5-30 人团队中负责主持会议和会后落地的人（项目经理、运营负责人、部门负责人）。首版按单用户工作流设计，不依赖全员注册。

### 4.2 你认为问题根因是什么？

根因不是工具不够多，而是会后行动项定义质量不达标：大量纪要只有“要做什么”，缺少“谁负责、何时完成、优先级”。这会让后续任何执行工具都失效。

### 4.3 如何衡量成功？

第一指标：会议结束 10 分钟内“关键字段完整率”（负责人+截止时间+优先级齐全）
目标：首批 20 家团队从基线 45%-55% 提升到 >= 85%。

### 4.4 与现有工具的差异？

不是任务管理平台，而是“会后质量闸门”：在任务进入聊天群或 PM 工具之前，先做行动项质检和字段补齐，保证输入质量。

## 5. Boundary 建议（锁定前直接用）

### In Scope

1. 中文会议纪要/文本导入
2. 行动项抽取
3. 关键字段缺失检测（负责人、截止时间、优先级）
4. 修复建议与一键补齐清单
5. 结果导出（文本/CSV）

### Out of Scope

1. 音频实时转写（首版仅文本导入）
2. 全团队协同账号体系
3. 站内推送中心和通知编排系统
4. 复杂项目排期与资源管理
5. 全量三方工具深度双向同步

### Open Questions

1. 首版导出格式优先 CSV 还是 Markdown
2. 首版是否支持自定义质检规则模板

## 13. 失败样本沉淀与最佳实践

从本次失败/波动数据提炼出的可执行实践：

1. LLM 输出需要 JSON 兜底层：真实模型经常输出非严格 JSON，必须有 parse fallback 与修复/降级策略。
2. Validation 结果与流程覆盖分离：产品评估结论可以是 no_go，但流程验收仍需有“覆盖后续阶段”的明确测试分支。
3. 外部 API 要有非阻断策略：GitHub 403/限流是常态，验收脚本要标记 skipped，而不是整条流程失败。
4. 每阶段都要写可观测日志：没有阶段日志很难定位卡点，尤其是 LLM 调用与外部 API 超时。

## 19. 本次 fatal risk 的真实根因与改写方向

本次 prosecutor/fatal 核心不是“实现能力不足”，而是价值链叙事冲突：

1. 你说解决“行动项缺失”，但输入前提仍依赖已有纪要，容易被质疑“真正痛点在无纪要/乱纪要”。
2. 使用者（会议主持人）与付费者（管理者/PMO）未打通，采购动机不充分。
3. 与“转写+行动项一体化工具”相比，差异点不够锋利。

改写原则（不改 Validation 机制）：

1. 明确产品定位为“会后执行质量闸门（QA）”，不是转写工具。
2. 明确输入来源是“任何已有文本源”（人工纪要、ASR 转写结果、群聊总结），而非只限人工纪要。
3. 明确付费对象是对交付负责且有预算权限的人（PMO/部门负责人）。
4. 明确差异化是“中文语境质量规则 + 可审计补齐链路 + 管理指标看板”。

## 20. 可直接替换的 Feed 输入（Anti-fatal 版）

我们做一个“会后行动项执行质量闸门（Action QA）”，面向 20-200 人、已要求会后留痕的团队。系统接收任何会后文本输入（人工纪要、语音转写文本、群聊总结），自动识别行动项并校验负责人、截止时间、优先级三项执行关键字段；对缺失项给出修复建议并生成可审计补齐记录与导出清单。

本产品不做语音转写，不替代任务管理系统，而是作为“进入执行系统前的质量闸门”。上线目标是帮助 PMO/部门负责人降低任务定义不完整导致的延期与返工。

首版范围：文本导入、行动项识别、缺失检测、一键补齐、导出与质量指标看板（关键字段完整率、会后补问时长）。不做团队协同账号、通知中心、复杂排期。

## 21. 手动验证时的关键答法（针对 fatal）

1. 为什么不是做转写？
	- 转写是上游能力，市场已有成熟供给；我们专注下游“执行质量闸门”，与现有转写/IM/PM 工具互补。
2. 谁付费？谁使用？
	- 使用者是会议主持人；付费决策者是 PMO/部门负责人，因为他们承担延期与返工成本。
3. 核心差异是什么？
	- 中文团队场景下的字段完整性规则、可审计补齐流程、以及面向管理者的质量指标闭环。
