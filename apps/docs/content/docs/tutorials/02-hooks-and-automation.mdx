## Hooks：让 Claude Code 守规矩的 8 条铁律

> 本章目标：能写一个 PreToolUse hook 拦截危险命令
> 解决的问题：Claude Code 经常乱删文件、改坏代码——怎么让它守规矩

### 1. Hooks 是什么

你有没有过这种经历——Claude Code 帮你改代码，改着改着，突然跑了个 `rm -rf node_modules`。

不是它坏。是它太听话。你让它"清理一下"，它真的清得干干净净。

Hooks 就是插在 CC 和你之间的安检门。每次 CC 要动手——不管是跑命令、写文件、还是调工具——hooks 先拦下来看一眼。可以放行、可以改、可以直接毙掉。

它不是 cron（定时触发），不是 Makefile（构建触发）。它是**事件驱动的拦截器**——CC 走到某个生命周期节点，hooks 自动跳出来。

### 2. 三张桌子，五把椅子

Hooks 的世界分两层：**什么时候拦**（事件），和**怎么拦**（handler）。

**三层事件**——从大到小：

| 层级 | 事件 | 干什么用 |
|---|---|---|
| 会话级 | SessionStart / Stop / Notification | 开场注入 context，散场打扫 |
| 轮次级 | UserPromptSubmit / SubagentStart | 你每次发话前先过安检 |
| 工具级 | PreToolUse / PostToolUse | **最重的一层**——每次调工具都拦 |

**五类 handler**——安检员的不同形态：

| Handler | 一句话 | 什么时候用 |
|---|---|---|
| `command` | 跑个 shell 脚本 | 简单判断，最快 |
| `http` | 调个外部 API | 需要查数据库、PR 状态 |
| `prompt` | 让 LLM 自己判断 | "这个操作合理吗" |
| `agent` | 派 subagent 判断 | 同上但更贵更准 |
| `mcp_tool` | 复用已有 MCP 能力 | 别重复造轮子 |

> 「那到底用哪个？」小李皱着眉。
>
> 「拦截 `rm -rf` 用 command，几十行 bash 搞定，」陈工说，「判断这个 PR 合进来会不会崩——那得调 CI，用 http。」
>
> 「prompt 呢？」
>
> 「贵。每次判断烧 token。只在你真需要 LLM 推理的时候用。」

关键记住：**PreToolUse 是唯一能改写 CC 输入的事件**——别的只能放行或拦住，PreToolUse 可以说"不许这么写，改一下"。

### 3. 8 条安全基线：不犯致命错的下限

hooks 有个要命的前提：**它以你系统用户的全权限跑**。能删文件、能读 `.env`、能把 secret 发出去。

这不是 bug，是设计。但你必须知道怎么防。

| # | 陷阱 | 一句防御 |
|---|---|---|
| 1 | 全权限跑 | hook 内部收敛权限 |
| 2 | 命令注入（`$VAR` 不加引号）| 永远 `"$VAR"` |
| 3 | 路径穿越（`../../`）| 校验必须在项目目录内 |
| 4 | 相对路径歧义 | 用 `${CLAUDE_PROJECT_DIR}` 绝对路径 |
| 5 | 直读 secret | 显式排除 `.env` / `.git` / `*.pem` |
| 6 | 无 timeout | 设 `timeout: N`，默认 60s |
| 7 | secret 写进日志 | 显式 redact |
| 8 | 无 controlling terminal | 通知走 `terminalSequence` 字段 |

(docs:L1)

记住一个原则：**白名单比黑名单安全**。你说"只允许 git、npm、ls"——这不会漏。你说"禁止 rm、sudo、curl"——总有你没料到的命令。

### 4. 实战：写一个 pre-bash-validator.sh

直接给模板。下面这个脚本，放到 `.claude/hooks/pre-bash-validator.sh`，挂上 PreToolUse：

```bash
#!/usr/bin/env bash
# stdin 收 JSON: { tool: "Bash", toolInput: { command: "..." } }
set -euo pipefail
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.toolInput.command')

# 白名单：只允许这些命令前缀
if [[ ! "$COMMAND" =~ ^(npm|yarn|pnpm|git|ls|cat|grep|find|echo|mkdir|cp|mv)\  ]]; then
  echo "BLOCK: 命令不在白名单中" >&2
  exit 2
fi

# 额外防线：即使白名单命令，也不许删根
if echo "$COMMAND" | grep -qE 'rm\s+-rf?\s+/'; then
  echo "BLOCK: rm -rf / 绝不允许" >&2
  exit 2
fi

exit 0
```

配到 `.claude/settings.json`：

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": ".claude/hooks/pre-bash-validator.sh"
      }]
    }]
  }
}
```

(docs:L1)

三条原则：
1. **白名单优先**——比黑名单少一个数量级的漏洞
2. **exit 2 才阻塞**——exit 1 只是警告，CC 继续跑；**只有 2 是真拦住**
3. **stderr 进 agent context**——CC 能看到拦截理由，下次会学乖

### 5. 企业级：一把锁管所有人

如果你带团队用 CC，有个场景躲不掉：

> 同事 A 写了个"方便"的 hook，从仓库读 `.env` 发到自己的调试服务器。
>
> 他不是故意的。他只是想快点修 bug。

`allowManagedHooksOnly` 就是干这个的。在 managed settings 里设一次：

```json
{ "allowManagedHooksOnly": true }
```

效果：**user 级、project 级、plugin 级的 hooks 全部失效**——只跑管理员配的 hook。

(docs:L1)

什么时候开：
- 公司 repo 有敏感数据（谁家没有）
- 团队超过 3 个人（信任不验证 = 不负责任）
- 合规要求（SOC2、ISO27001——审计员会问"你们的 AI 工具能删生产数据吗"）

### 6. 横向对比：CC Hooks 不是唯一的拦截器

| 维度 | CC Hooks | Cursor /review | Codex Hooks | GitHub Actions |
|---|---|---|---|---|
| 触发点 | 生命周期内 9 个事件 | 推送前 / PR 同步 | 10 event | workflow 文件 |
| 跑在哪 | 你本地 | 云端 / 浏览器 | 本地 Rust | 云端 runner |
| 能改写输入 | ✅ PreToolUse | ❌ | ✅ (部分) | ✅ (环境变量) |
| 企业强制 | ✅ allowManagedHooksOnly | RBAC | approval policy | org secrets |

(docs:L1)

简单说：**CC Hooks 是本地最细颗粒度的拦截**——GitHub Actions 管 CI/CD，CC Hooks 管"这一刻 CC 想干什么"。两把锁，不在同一层。

### 7. 本章小结 + 下一章预告

**你带走的：**
- Hooks = CC 生命周期的安检门——不是 cron，不是 Makefile
- PreToolUse 是唯一能改写操作的事件——别的只能放行或拦住
- 8 条安全基线里，**白名单 &gt; 黑名单** 是最核心的一条
- `pre-bash-validator.sh` 模板拷走就能用——改白名单那行即可
- 团队 &gt; 3 人，`allowManagedHooksOnly` 是标配不是可选项

**下一章**：Hooks 让 CC 守规矩——但规矩只管一次会话。明天开新 session，CC 又不认识你了。第三章讲**记忆系统**——怎么让 CC 跨会话记住你是谁、你们团队的规范、你的偏好。
