你的贪吃蛇能玩了。你最高分 92。

朋友来家里，玩了一把——87 分。你说："我比你高。"他说："证明给我看。"你刷新了页面——分数没了。

游戏做出来了。但除了你，没人知道你拿了 92 分。

## 加上本地排行榜

让 CC 给贪吃蛇加一个本地排行榜——分数存在浏览器里，刷新不丢。用 localStorage。

> add feature：游戏结束时弹出输入框让玩家输入昵称，然后把分数 + 昵称存到 localStorage。同时在游戏界面上方显示历史 Top 10 排行榜。按 add-snake-feature Skill 的步骤来——只改该改的部分。

如果 CC 按 Skill 流程走——它应该在 `main.ts` 里加：配置区（`MAX_LEADERBOARD_SIZE = 10`）、状态区（`leaderboard: ScoreEntry[]`）、逻辑区（`loadLeaderboard` / `saveScore` 函数读写 localStorage）、渲染区（排行榜 UI 列表）。不该动蛇的移动逻辑和食物生成。

跑起来。玩一局。刷新页面——分数还在。

然后你会发现一个新问题。

## 第一堵墙：CC 不记得你的项目规范

排行榜加好了。你又想加一个新功能——"食物偶尔出现金色星星 ⭐，吃到 +50 分"。

你关了昨天的 session。新开了一个。说：

> add feature：食物有 20% 概率是 ⭐，吃到 +50 分

CC 改了代码。`tsc --noEmit` 也过了。但跑起来——蛇头变成了方块，背景色变了，排行榜字体从红色变成了白色。

你回去看代码。CC 重新定义了 `GameState` 接口。它不知道你的蛇头是 🐍，食物是 🍎，得分颜色是 `#e94560`。这些规矩你上次口头告诉过它——但那是**上个 session**。CC 全忘了。

这不是 CC 的 bug。CC 的每个新 session 都是白纸一张——除非你给它写下来。

## Memory：用 CLAUDE.md 让 CC 跨 session 记住规矩

在 VS Code 中创建 `CLAUDE.md`：

```markdown
# 贪吃蛇项目规范

## 视觉规范
- 蛇头用 🐍 emoji 绘制
- 食物用 🍎 emoji（普通）或 ⭐ emoji（金色星星，20%概率，+50分）
- 得分显示颜色 #e94560，字体 24px bold
- Canvas 400×400，20px 一格（20×20 网格）

## 游戏参数
- 初始移动速度 150ms 一步
- 分数 = 蛇身长度 × 10（普通食物）或 +50（金色星星）
- 排行榜保留 Top 10，存在 localStorage key `snake-leaderboard`

## 代码规范
- TypeScript strict mode
- 单文件 src/main.ts
- 配置常量放文件顶部
- 渲染逻辑在 draw 函数，游戏逻辑在 update 函数
- 每次改动后跑 `npx tsc --noEmit`（pnpm：`pnpm exec tsc --noEmit`）

## 禁止事项
- 不要改 Canvas 大小
- 不要改游戏循环的 requestAnimationFrame 结构
- 不要引入额外 npm 包
```

保存。告诉 CC：

> 读一下 CLAUDE.md。这是这个项目的规范。以后每次你改代码，先检查改动是否违反规范。

现在验证：关掉这个 session。开新的。问：

> 我们这个贪吃蛇项目用什么语言？蛇头是什么 emoji？食物有几种？

如果 CC 答出 TypeScript、🐍、两种食物——Memory 生效了。

### CLAUDE.md 是怎么工作的

CC 在每次 session 启动时自动读取项目根目录的 `CLAUDE.md`，把它作为**指南**注入到上下文中。注意是"指南"不是"强制"——CC 会尽量遵守，但不保证。如果要硬性拦截某些操作，后面讲 Hooks 时会说到。

几个关键点（从官方文档和实践中总结的）：

1. **CLAUDE.md 放在项目根目录**——这是 CC 默认读取的位置。也可以放在 `.claude/CLAUDE.md`，效果一样。
2. **多项目怎么办？** `~/.claude/CLAUDE.md` 是用户级的——所有项目通用。项目级覆盖用户级。
3. **写什么？** 那些你每次开 session 都要口头重复一遍的东西——技术栈、命名规范、禁止事项。
4. **不要写什么？** 不要写"好好写代码"、"注意性能"——这些太模糊，CC 无法执行。写具体的：蛇头是 🐍、Canvas 是 400×400。

## 第二堵墙：CC 改完代码，类型炸了

排行榜跑了几天。你又加了一个功能——"蛇撞到自己时闪烁三下再 game over"。

CC 改了代码。`tsc --noEmit` 也跑了——但你忘了看输出。你直接保存、刷新页面——白屏。

CC 在 `GameState` 接口加了一个 `flashCount` 字段。但它在 `update` 里用这个字段，却没有在 `initState` 里初始化。TypeScript 报了 `Property 'flashCount' does not exist on type 'GameState'`。你错过了。

## Hooks：让 CC 自己查自己的活

Hooks 是 CC 的"自动检查"机制——在 CC 做某个操作之前或之后，自动跑一段你指定的动作。跟 Git hooks 一个思路：`pre-commit` 跑 lint，`post-merge` 跑 install。

你要的是：**CC 每次改完 TypeScript 文件，自动跑类型检查。没过 = 不算改完。**

在 VS Code 中创建 `.claude/hooks/post-tool-use.md`：

```markdown
# PostToolUse Hook：TypeScript 类型自检

当 CC 使用 Write 或 Edit 工具修改了 .ts 文件后：

1. 自动执行 `npx tsc --noEmit`（pnpm：`pnpm exec tsc --noEmit`）
2. 如果 tsc 报错——列出报错文件和行号，告诉用户"类型检查未通过，代码可能有问题。不主动修——等用户说修再修。"
3. 如果 tsc 通过——回复末尾加 `✅ 类型检查通过`
```

这个 Hook 的含义：**改 .ts → 跑 tsc → 告诉用户结果。** CC 改代码的动作和验证代码的动作绑在了一起。

现在再让 CC 加一个功能——"蛇身超过 10 节后速度提升 20%"：

> add feature：蛇身超过 10 节后，移动步长自动缩短 20%

这次注意看 CC 改完后的回复——末尾有没有 `✅ 类型检查通过`。如果没有——提醒它："你的 Hook 说改完 .ts 文件要跑 tsc。"

### Hooks 是怎么工作的

Hooks 在 CC 的"生命周期关键节点"自动触发。你刚用的是 PostToolUse——在 CC 使用工具（如 Write、Edit、Bash）**之后**触发。

CC 的 Hooks 分为几个层级：

| 层级 | 触发时机 | 典型用途 |
|---|---|---|
| **会话级** | SessionStart / SessionEnd | 注入上下文 / 清理临时文件 |
| **轮次级** | UserPromptSubmit | 输入验证 |
| **工具级** | PreToolUse / PostToolUse | 权限闸口 / 自动验证 |

PreToolUse 在 CC **执行操作前**触发——可以拦截危险操作（如 `rm -rf`、`DROP TABLE`）。PostToolUse 在**执行后**触发——适合自动验证（如你的 `tsc --noEmit`）。

一个重要的区分：**CLAUDE.md = 指南（CC 尽量遵守），Hooks = 硬约束（CC 无法绕过）。** 如果一条规则绝对不能违反——用 Hook 不用 CLAUDE.md。

对 Hooks 的完整深入——第 05 章专门讲。现在你手里的够用了。
