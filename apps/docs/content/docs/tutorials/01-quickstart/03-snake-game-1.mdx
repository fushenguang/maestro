你打开 CC，说："做个贪吃蛇。"

CC 10 秒生成了代码——能跑。蛇在屏幕上爬，方向键能转弯。

你兴奋了。"再加个排行榜！"CC 又生成了代码——但这次蛇不动了，排行榜也没出来，Canvas 大小变了。

你意识到：告诉 CC **怎么做**不是靠每次口头说。是靠写下来。

## 先让它跑起来

```bash
mkdir snake-game && cd snake-game
npm init -y
npm install typescript vite --save-dev
npx tsc --init --target ES2020 --module ESNext --moduleResolution bundler --strict
```

<details>
<summary>如果你用 pnpm</summary>

```bash
mkdir snake-game && cd snake-game
pnpm init -y
pnpm install typescript vite --save-dev
pnpm exec tsc --init --target ES2020 --module ESNext --moduleResolution bundler --strict
```

pnpm 会在 `package.json` 注入 `devEngines.packageManager: "pnpm"`，之后用 `npx` 会报 `EBADDEVENGINES`（name 不匹配）。全书统一：pnpm 用户用 `pnpm exec` 替代 `npx`。

</details>

用 VS Code 打开项目，创建 `index.html`：

```bash
code . -r
```

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>贪吃蛇</title>
  <style>
    body { margin: 0; display: flex; justify-content: center; align-items: center; min-height: 100vh; background: #1a1a2e; font-family: system-ui; }
    #game { display: flex; flex-direction: column; align-items: center; gap: 12px; }
    canvas { border: 2px solid #e94560; border-radius: 4px; }
    #score { color: #e94560; font-size: 24px; font-weight: bold; }
  </style>
</head>
<body>
  <div id="game">
    <div id="score">分数：0</div>
    <canvas id="canvas" width="400" height="400"></canvas>
  </div>
  <script type="module" src="/src/main.ts"></script>
</body>
</html>
```

**不要自己写 `main.ts`——让 CC 来。** 把下面这段话贴进 CC：

> 帮我写一个 TypeScript 贪吃蛇游戏，文件放在 src/main.ts。
> 要求：
> - Canvas 2D 渲染，400×400，20px 一格（20×20 网格）
> - 方向键/WASD 控制方向
> - 吃到食物蛇身增长 1 格，分数 +10
> - 撞墙或撞到自己 = game over，alert 弹窗显示分数
> - 游戏循环用 requestAnimationFrame，初始速度 150ms 一步
> - 所有代码写在一个文件里

CC 会生成 `src/main.ts`。在 VS Code 中保存（`Ctrl+S` / `Cmd+S`）。然后编辑 `package.json`，加 `scripts`：

```json
// package.json 加 scripts
{
  "scripts": {
    "dev": "vite"
  }
}
```

```bash
npm run dev
```

打开 `http://localhost:5173`——蛇在爬。你做出了第一个东西。耗时大概 10 分钟。

## 然后你撞上墙了

玩了两把，你想："蛇头能不能用 emoji？🐍"

你在 CC 里说：

> 帮我把蛇头改成 🐍 emoji。

CC 生成了新代码。蛇头是 emoji 了。但速度变了，Canvas 变成了 500×500，分数框的样式丢了。

回想一下：每次你让 CC 加新功能，它都**重新生成了整个文件**。像每次装修都拆了重盖——换一盏灯，结果整面墙被推倒了。

这不是 CC 的错。是你没给它一个规矩。

## Skill：把"加功能的规矩"写进文件

回想第 02 章的核心教训：**你说得越模糊，CC 猜得越多。改得越多。** 加新功能也是同理——如果你不告诉 CC "加功能的固定流程"，它每次自己发明流程。每次都拆了重盖。

Skill 文件就是那个固定流程。在 VS Code 中创建 `.claude/skills/add-snake-feature/SKILL.md`：

```markdown
---
name: add-snake-feature
description: "当用户说 add feature [功能描述] 时，按固定流程在贪吃蛇项目中加新功能。MUST 按步骤来——只改该改的部分，不重构整个文件。"
---

# 给贪吃蛇加一个新功能

## 1. 理解需求
- 把用户的功能描述拆成三部分：**影响什么配置** / **改什么逻辑** / **改什么渲染**

## 2. 修改代码（只改需要的，不动结构）
- **main.ts 的骨架不要动**——只在你需要的地方加代码
- 2a. 配置区（文件顶部常量）：只加新常量，不改已有的值
- 2b. 状态区（GameState 接口）：只加新字段，不改已有字段
- 2c. 逻辑区（update 函数）：只加新逻辑分支，不改旧逻辑
- 2d. 渲染区（draw 函数）：只加新绘制调用，不改旧绘制

## 3. 验证
- 改完后跑 `npx tsc --noEmit`（pnpm 用户用 `pnpm exec tsc --noEmit`），确认类型通过
- 如果报错——修类型，不改逻辑

## 4. 报告
- 告诉用户：改了什么文件、什么位置、加了多少行、删了多少行
- 格式：`改了 src/main.ts：+15 行，-2 行。类型检查通过。`
```

注意这个 Skill 的结构——它不是一个"功能描述文档"，是一个**标准操作流程（SOP）**。好的 Skill 有三个特征：
1. **触发条件明确**（`description` 写了"当用户说 add feature 时"——这不是摘要，是触发器）
2. **步骤具体到操作**（不是"处理渲染"而是"只加新绘制调用，不改旧绘制"）
3. **有验证步骤**（改完跑 `tsc --noEmit`——不是建议，是流程的一部分）

存好文件。然后告诉 CC：

> 看一下 .claude/skills/add-snake-feature/SKILL.md。以后我每次说 "add feature xxx" 时，按这个 Skill 的步骤来。

CC 会读那个文件，记住流程。现在再试：

> add feature：蛇头用 🐍 emoji 绘制

这次 CC 应该**只改 draw 函数里的蛇头绘制部分**，不动其他地方。改完自动跑 `tsc --noEmit`。

如果它确实只改了渲染——你刚写出了第一个有效的 Skill。

如果它又重写了整个文件——别急。提醒它：

> 你违反了自己的 Skill——add-snake-feature SKILL.md 第 2 条说"main.ts 的骨架不要动"。回滚，只改 draw 里的蛇头绘制。

## 你刚刚学会了什么（不止贪吃蛇）

Skill 的本质不是"让 CC 做更多"。是**让 CC 每次都做一样的事**。

没有 Skill：你说"加排行榜"→ CC 重新发明一次流程 → 结果不可预测 → 你每次都要口头教。

有 Skill：你说"add feature 排行榜"→ CC 按固定 SOP 走 → 结果可预测 → 你教了一次，记在文件里。

跟管理一个新同事一模一样。第一周不给 checklist——每次自由发挥，每次闯祸。第二周给了 checklist——照着做，产出稳定。Skill 就是 CC 的 checklist。

## 验证你的 Skill：试三个功能

用 `add feature xxx` 格式，依次加：

1. **食物用 🍎 emoji 绘制**——验证只改渲染（2d），不碰逻辑
2. **蛇身每节用不同深浅的绿色**——验证逻辑+渲染（2c+2d），不碰配置
3. **按空格键暂停/继续**——验证配置+状态+逻辑（2a+2b+2c），不碰渲染

三次都是"只改该改的 + `tsc --noEmit` 通过"——你的 Skill 及格了。

如果某次又"重构了整个文件"——回去改 SKILL.md。第 2 条的约束不够强？加一句：

> ⚠️ 最高原则：改动前，先在回复里列出你要改哪些部分，征得我同意后再动手。如果某个功能只需要改渲染——你不许碰逻辑和状态。如果只需要改逻辑——你不许碰渲染。

这就是 Skill 开发的真实过程：写 → 试 → 发现漏洞 → 改进 → 再试。第 07 章会深入讲怎么系统地写和评估 Skill。

## 为什么先不学 Hooks、Memory、MCP

因为你现在不需要。

你现在有一个会爬的蛇、一个能加功能的 Skill 模板。**做出来是第一步。** 下一章你会给它加排行榜，让朋友能跟你比分数。到时候你会撞上新问题——CC 改了代码但忘了做类型检查，或者每次新开 session 都要重新告诉它"蛇头是 emoji"。那时候你自己会想知道 Hooks 和 Memory 是什么。

但现在——先玩两把你做的贪吃蛇。
