你有一个 TypeScript 项目。一个函数老是返回 `undefined`。你打开 CC，把代码贴进去："帮我修一下这个 bug。"

CC 看了几秒。说"问题在于你的类型定义不够严格"——然后开始重写你的整个路由层。

你吓得按了 `Ctrl+C`。

这不是 CC 的 bug。这是它的设计：CC 默认假设你要的是**最好的结果**，不是**最小的改动**。像一个极度勤奋的新同事——能力强，但不知道边界在哪。

这一章教你画边界。

## 准备一个可以随便折腾的项目

新建一个 playground：

```bash
mkdir cc-playground && cd cc-playground
npm init -y
npm install typescript --save-dev
npx tsc --init
```

<details>
<summary>如果你用 pnpm</summary>

```bash
mkdir cc-playground && cd cc-playground
pnpm init -y
pnpm install typescript --save-dev
pnpm exec tsc --init
```

pnpm 用户全书统一用 `pnpm exec` 替代 `npx`，避免 `devEngines.packageManager` 冲突。

</details>

用 VS Code 打开项目，创建有 bug 的文件 `src/utils.ts`：

```bash
code . -r
```

```typescript
// 这个函数有一个 bug
export function getUserName(user: { name?: string } | null): string {
  return user.name.toUpperCase();
}
```

`user` 可能是 `null`，`user.name` 可能是 `undefined`。调用就炸。

现在进 CC：

```bash
claude
```

## 控制 CC 的三个手段

### 手段 1：把话说精确

CC 对模糊指令的处理方式是"猜"。猜得越多，改得越多。对比：

| 模糊的说法 | CC 可能做的事 |
|---|---|
| "修一下这个 bug" | 重写整个文件 + 重构类型系统 + 建议装 zod |
| "优化一下性能" | 重写整个渲染层 + 引入 React.memo + 改打包配置 |
| "帮我加个功能" | 新建三个文件 + 改五个已有文件 + 更新 package.json |

| 精确的说法 | CC 做的事 |
|---|---|
| "在 `getUserName` 函数第 3 行，`user.name` 前加 `user?.`，不改其他代码" | 改一行 |
| "把 `UserCard` 组件的 3 个 `useState` 合并成一个 `useReducer`，不动 props 接口" | 改一个组件 |
| "在 `src/utils.ts` 加一个 `formatDate` 函数，输入 Date，返回 `YYYY-MM-DD` 字符串" | 加一个函数 |

**原则**：你说不清楚改哪里 → 先自己搞清楚 → 再说。这一条省你未来无数个 `Ctrl+C`。

### 手段 2：权限弹窗——你的最后防线

CC 执行"可能有影响"的操作前会弹确认：

```
⏺ CC wants to run:
  rm -rf ./node_modules

Allow? (y)es / (n)o / (a)lways allow
```

| 选项 | 含义 | 什么时候用 |
|---|---|---|
| `y` (yes) | 这次允许，下次还问 | 偶尔用一次的操作 |
| `n` (no) | 这次不允许 | 你觉得 CC 做错了——点 `n`，然后告诉它为什么 |
| `a` (always allow) | 永远允许，不再问 | 你完全信任这个操作模式 |

**前两周，永远不要点 `a`。** 等你摸清了 CC 什么时候会"过度发挥"、什么时候真的只是装个依赖——再决定哪些操作可以永远放行。特别是 Bash 类的权限（`rm`、`mv`、`chmod`、`sudo`）——先手动确认两周。这条建议值你未来少踩十个坑。

### 手段 3：Git——终极撤销键

CC 改的文件在 Git 仓库里随时可以回滚：

```bash
# 看 CC 改了什么
git diff

# 回滚单个文件
git checkout -- src/utils.ts

# 回滚所有未提交的改动（⚠️ 你自己的改动也会丢）
git checkout -- .
```

**如果你的项目还没用 Git——现在立刻：**

```bash
git init && git add . && git commit -m "init"
```

这句话值你未来所有被 CC 改坏的代码。

## 三个实战场景

### 场景 A：只分析，不改代码

CC 没有"只读模式"。让它不动手的唯一方式是**在 prompt 里说清楚**：

```
看一下 src/ 目录的文件结构。告诉我哪个文件代码质量最差，但不要改任何东西、不要运行任何命令。
```

把约束放在句末——"不要改任何东西"——CC 会遵守。

如果它还是想动手——权限弹窗弹出来的时候点 `n`，然后提醒它："我说了只看不改。"

### 场景 B：改指定的东西，不动其他的

```
在 src/utils.ts 里：
1. 给 getUserName 加 null 检查（第 3 行 user.name → user?.name）
2. 其他代码一行不改
3. 改完跑 `npx tsc --noEmit`（pnpm：`pnpm exec tsc --noEmit`），确认类型通过
```

这个 prompt 的骨架：**指定文件 + 指定位置 + 指定操作 + 验证步骤。** CC 没有空间"发挥"——每一步都框死了。

### 场景 C：CC 改完代码，类型炸了

你让 CC 改了东西。它说"改好了"。你刷新页面——白屏。因为它改了 `GameState` 接口但没在初始化函数里加对应字段。

养成习惯：**CC 说改好了 → 先跑类型检查 → 再刷新页面。** 让 CC 自己跑：

```
改完了？跑一下 `npx tsc --noEmit`（pnpm：`pnpm exec tsc --noEmit`），确认没有类型错误。
```

如果 CC 改完自动跑了且通过了——你会看到输出末尾的 `✅ 类型检查通过`。如果没有——主动让它跑。后面第 04 章会教你怎么让这一步自动化（用 Hooks）。

## 几个真正有用的命令

CC 的内置命令不多，但这几个值得记住：

| 命令 | 作用 | 什么时候用 |
|---|---|---|
| `/clear` | 清空当前对话上下文 | CC 开始"记混了"、重复之前的错误 |
| `/compact` | 压缩上下文（保留关键信息）| 对话太长，CC 开始变慢或健忘 |
| `/model` | 切换模型 | 简单问题用便宜的模型，复杂问题用强的 |
| `/memory` | 查看当前加载的 CLAUDE.md 和 rules | 你不确定 CC 有没有读到你的项目规范 |

没有 `/prompt` 这个命令——你在终端里打字就是 prompt，不需要前缀。

## 自检

把你的 `cc-playground` 项目打开。用 CC 把 `getUserName` 的 bug 修好——**只改一行代码**（`user.name` → `user?.name`），不改其他任何东西。

做完后：`git diff`——确认只改了一行。

做到了？下一章。

> **留白**：记下来——你跟 CC 说了几次它才只改了一行？如果超过 3 次——你的 prompt 还不够精确。这是正常的。第 18 章会回来专门讲 Prompt 工程。
