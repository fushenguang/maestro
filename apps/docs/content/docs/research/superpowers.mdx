# Superpowers

## Card（速览）

### 一句话定位

> 一套可组合的「Skills」框架 + 软件开发方法论，让 Claude Code / Codex / Cursor 等 AI 编码代理自动按正确流程工作。

### 关键 takeaway

1. **Skill 本质是一个 `.md` 文件 + frontmatter**，描述「在什么情况下自动触发 → 要做什么」。`SKILL.md` 是核心交付物。
2. **7 步核心工作流**：`brainstorming` → `using-git-worktrees` → `writing-plans` → `subagent-driven-development` (或 `executing-plans`) → `test-driven-development` → `requesting-code-review` → `finishing-a-development-branch`。
3. **强制而非建议**：agent 启动即检查相关 skill，user 不必手动调用——这是它和「命令式插件」最本质的区别。
4. **TDD 是底线**：`test-driven-development` 强制 RED-GREEN-REFACTOR 循环 + 反模式库（`testing-anti-patterns.md`）；写代码前先写测试，测试前先看它红。
5. **跨 7 个 coding agent**：Claude Code、Codex CLI / App、Cursor、OpenCode、Gemini CLI、GitHub Copilot CLI、Factory Droid——同一份 skill 7 处可用。

### 与 Claude Code 教程的关联

- 团队已通过 `/plugin install superpowers@claude-plugins-official` 装上 superpowers，本次「为 Claude Code 写教程」的工作流本身就是它（brainstorming → writing-plans → subagent-driven-development）的实践产物，**最有说服力的对照案例**。
- 教程应展示：**「我们自己写一个 Skill 是什么样的」**——SKILL.md 的 frontmatter 结构、触发条件描述、references 目录组织、scripts/ 配套。可以拿本仓库的 `openspec/AGENTS.md` 和本 Skill（`research-source`）作为本地反例。
- 进阶章节可对比「superpowers 的 skill」 vs「Claude Code 原生的 `/skill` 命令 + 自定义 skill」——两者的能力模型差异（自动触发 vs 显式调用）是教程必讲的关键。

## Deep-Dive（深读）

### SKILL.md 的解剖（以 `brainstorming` 为样板）

```yaml
---
name: brainstorming
description: "You MUST use this before any creative work..." 
---
```

**Frontmatter 三个核心字段**：

| 字段 | 必填 | 职责 | 写法要点 |
|---|---|---|---|
| `name` | 是 | kebab-case 标识 | 与目录名一致 |
| `description` | 是 | **触发器不是摘要** | 写「在什么请求下激活我」+ 触发关键词 + 强制语气（"MUST" / "use this before"） |
| 其它（`allowed-tools` 等） | 否 | 由 harness 决定 | 参考各 harness 文档 |

### Body 必备的 5 个段

| 段 | 职责 | 反例 |
|---|---|---|
| **Hard Gate** | 明确禁止跳过该 skill 的边界条件 | 「简单项目可以跳过」 |
| **Anti-Patterns** | 列出常见错误 + 后果 | 「这个项目太简单不需要设计」 |
| **Checklist** | 步骤清单（顺序、强约束） | 没有顺序感的散文 |
| **Process Flow** | 用图（dot/mermaid）表示分支 | 纯文字描述 |
| **Spec Section** | 产出物标准（写到哪、含什么） | 模糊「写个文档」 |

**实际例子**（brainstorming 顶部）：
- HARD-GATE：禁止 implementation 行动直到 design 被 user 批准
- Anti-Pattern："This Is Too Simple To Need A Design"——简单项目也必须过 design
- 9 步 Checklist：Explore context → Visual companion offer → Clarifying questions → 2-3 approaches → Present design → Write spec → Self-review → User review → Writing-plans

### 文件夹组织（progressive disclosure）

```
skills/my-skill/
├── SKILL.md              # 入口 frontmatter + 主体
├── references/           # 详细文档（按需读，不一次全加载）
│   ├── api.md
│   ├── gotchas.md
│   └── examples.md
├── scripts/              # 可执行代码（Claude 可调用）
│   ├── helper.py
│   └── verify.sh
└── assets/               # 模板、配置等
    └── template.md
```

**为什么用文件夹**：
- 大 skill 一次全塞 SKILL.md 会撑爆 context
- 引用文件 + 给 Claude 描述「何时读什么」= 渐进式信息披露
- 配合 scripts：给 Claude 工具而不是给 Claude 文档——「Store scripts and generate code」（claude-skills-blog 第 8 条 tip）

### writing-skills 元流程（5 步）

`writing-skills/SKILL.md`（655 行，最大 SKILL.md）规定：

1. **写第一版**——最小可用 + 1 个 gotcha
2. **跑对抗性测试**——多次会话，red team 找漏洞
3. **改 skill**——按 gotchas 增补
4. **A/B 评估**——前/后对比，给出 eval evidence（这是 CLAUDE.md 强制要求，没有 eval 的 skill 修改会被拒绝）
5. **贡献**——如果值得共享，参考 [Anthropic 官方 9 大分类](https://claude.com/blog/lessons-from-building-claude-code-how-we-use-skills)（同站 `claude-skills-blog`）

### 教程可直接复用的 4 件事

1. **「写一个 Skill」checklist** = writing-skills 第 1-5 步简化版（300 字内）
2. **SKILL.md 5 段模板** = 直接给读者一个填空骨架
3. **Description 写作指南** = "You MUST use this before..." 模板 + 触发关键词清单
4. **9 大分类决策树**（参考 claude-skills-blog）：读者想「我要写一个 skill」→ 先问「它属于哪一类？」
