# Memory 与 CLAUDE.md（官方文档）

## 一句话定位

> Claude Code 的「跨会话知识」由两套互补机制承担：CLAUDE.md（人写的持久指令，4 层加载）+ auto memory（Claude 自己累积的学习笔记）；两者都在每场会话开始时进入 context，但都是「指南」而非「强制」——硬约束要用 hook。

## 关键 takeaway

1. **CLAUDE.md ≠ auto memory，两套都加载**：CLAUDE.md 由人写、范围 project / user / org、整段注入；auto memory 由 Claude 写、范围 per-repo、只截 MEMORY.md 前 200 行或 25KB 注入。两者作为 user message 进入 context（**不是** system prompt），不强制——硬阻断动作用 `PreToolUse` hook。锚点 `docs:L1:14, L1:1736`。
2. **CLAUDE.md 4 级加载链**（从广到窄）：**Managed policy**（org 级，IT 部署，不可个人 exclude）> **User**（`~/.claude/CLAUDE.md`）> **Project**（`./CLAUDE.md` 或 `./.claude/CLAUDE.md`，进版本控制）> **Local**（`./CLAUDE.local.md`，**必须 gitignore**）；多文件按目录树「从根到 cwd」**concatenate** 而非 override。锚点 `docs:L1:80, L1:10237`。
3. **子目录懒加载 + HTML 注释剥离**：cwd **之上** 的 CLAUDE.md 启动即注入；cwd **之下** 的子目录 CLAUDE.md 仅在 Claude 读取该子目录文件时才挂入——monorepo 噪声控制的关键。`<!-- maintainer notes -->` 这种块级 HTML 注释在注入前被剥掉（code 块内保留）。锚点 `docs:L1:11179, L1:11459`。
4. **`.claude/rules/` 拆分大文件 + `paths:` 域**：`testing.md` / `api-design.md` 等小文件加 `paths: src/api/**/*.ts` glob 后，**仅在 Claude 处理匹配文件时**挂入；不加 paths 则全量加载。配合 `@path/to/import` 跨文件引用（递归深度上限 4 跳）+ `~/.claude/rules/` 用户级规则 + AGENTS.md 桥接（`@AGENTS.md` 或 symlink）。锚点 `docs:L1:172, L1:196, L1:312, L1:13872`。
5. **auto memory 存 `~/.claude/projects/<project>/memory/`**：所有 worktree 与子目录共享同一 memory dir；MEMORY.md 当索引，前 200 行 / 25KB 注入。详细 topic 文件不预载，按需 Read。可用 `autoMemoryEnabled: false` 或 `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` 关闭，`autoMemoryDirectory` 可改路径。**机器本地，不跨机同步**。锚点 `docs:L1:551, L1:19465, L1:19364`。
6. **`/memory` 命令 + `/compact` 行为差异**：`/memory` 列当前 session 加载的所有 CLAUDE.md / rules + 切 auto memory 开关。`/compact` 后**根目录 CLAUDE.md 重新读盘并 re-inject**，**子目录 CLAUDE.md 不会自动 re-inject**——「compact 后某指令消失」基本是这两种情况之一。锚点 `docs:L1:591, L1:24563`。

## Brief（梳理）

### Memory 双层机制

- **CLAUDE.md = 人写、整段注入**（指南；硬约束要用 hook）
- **auto memory = Claude 写、按行截断注入**（学习笔记）
- **两者都进 context**（user message，**不是** system prompt）
- 都不强制——硬阻断动作用 `PreToolUse` hook

### CLAUDE.md 4 级加载链（从广到窄）

```
Managed policy  >  User  >  Project  >  Local
（IT 部署）         (~/.claude/)  (./CLAUDE.md)  (./CLAUDE.local.md)
                                   提交 git      gitignore
```

- **多文件**：从根到 cwd **concatenate**（不是 override）
- **Managed policy 不可个人 exclude**——IT 部署的最高级

### 子目录懒加载 + HTML 注释剥离

- **cwd 之上**：启动即注入
- **cwd 之下**（子目录 CLAUDE.md）：**仅在 Claude 读该子目录文件时挂入**——monorepo 噪声控制关键
- `<!-- maintainer notes -->` 这种 HTML 注释在注入前被剥掉（code 块内保留）

### `.claude/rules/` 拆分大文件

- `testing.md` / `api-design.md` 等小文件加 `paths: src/api/**/*.ts` glob 后，**仅在 Claude 处理匹配文件时挂入**
- 不加 `paths` 则全量加载
- **跨文件引用**：`@path/to/import`（递归深度上限 4 跳）
- **AGENTS.md 桥接**：`@AGENTS.md` 或 symlink

### auto memory 存哪

- `~/.claude/projects/<project>/memory/`（机器本地，**不跨机同步**）
- 所有 worktree 与子目录共享同一 memory dir
- **MEMORY.md 是索引**（前 200 行 / 25KB 注入）
- 详细 topic 文件不预载，按需 Read
- 关闭：`autoMemoryEnabled: false` 或 `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1`
- 改路径：`autoMemoryDirectory`

### `/memory` vs `/compact` 行为差异

| 命令 | 行为 |
|---|---|
| `/memory` | 列当前 session 加载的所有 CLAUDE.md / rules + 切 auto memory 开关 |
| `/compact` 后 | **根目录 CLAUDE.md 重新读盘 re-inject**；**子目录 CLAUDE.md 不会自动 re-inject** |

> 「compact 后某指令消失」基本是这两种情况之一

### 与其他源的关系

- **sub-agents**（`memory: user / project / local` 字段）：sub-agent 自己的 memory，与本节 CLAUDE.md / auto memory 是**两套并存**
- **Agent Skills**：按需挂载（与 CLAUDE.md 加载时机不同）
- **Settings**：`autoMemoryEnabled` / `claudeMdExcludes` 控制开关
- **hooks**：本节是「指南」、hook 是「强制」——核心区分

### 实战组合（推荐默认配置）

```text

项目 CLAUDE.md         → 工程规约（commit 进 git）

auto memory           → 沉淀踩坑笔记
```

## Deep-Dive（深读）

### 主题 1：双层机制 4 维对比（CLAUDE.md vs auto memory）

CLAUDE.md 和 auto memory 都是「跨会话知识」，但**作者不同、加载方式不同、约束力不同**——混用就翻车。这节把 Brief 已说的概念**精确化到字段级**。

| 维度 | CLAUDE.md | auto memory |
|---|---|---|
| **谁写** | 人（developer / IT / 项目维护者）| Claude（自动）|
| **范围** | org / user / project / local | per-project（per worktree 共享）|
| **存储路径** | `~/.claude/CLAUDE.md` / `./CLAUDE.md` / `./.claude/CLAUDE.md` / `./.claude/CLAUDE.local.md` | `~/.claude/projects/<project>/memory/` |
| **加载方式** | 整段 / concatenate / 子目录懒加载 | MEMORY.md 索引（前 200 行 / 25KB）+ topic 文件按需 Read |
| **注入位置** | user message（**不是** system prompt）| user message（**不是** system prompt）|
| **约束力** | 指南（model 可不遵守）| 指南（model 可不遵守）|
| **可关闭** | `claudeMdExcludes`（settings）| `autoMemoryEnabled: false` / `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` |
| **跨机同步** | ✅（commit 进 git）| ❌（机器本地）|
| **企业管控** | managed policy（IT 部署）| ❌ |

- **核心区分**：两者都是「指南」非「强制」——硬约束要 `PreToolUse` hook
- **核心互补**：人写「应知」（CLAUDE.md）+ Claude 写「所学」（auto memory）= 双源

### 主题 2：4 级加载链 + 子目录懒加载 + path-scoped rules 三件套

CC 的「加载机制」分三层抽象——每层解决不同问题：

#### 2.1 4 级加载链（Managed > User > Project > Local）

```
Managed policy    (org 级 / IT 部署 / 不可个人 exclude)
  ↓
User              (~/.claude/CLAUDE.md)
  ↓
Project           (./CLAUDE.md 或 ./.claude/CLAUDE.md)
  ↓                 (commit 进 git)
Local             (./.claude/CLAUDE.local.md)
                    (必须 gitignore)
```

- **concatenate 而非 override**——多文件按目录树「根到 cwd」**全部加载**（不是后置覆盖前置）
- **Managed policy 不可 exclude**——IT 部署的最高级（企业安全基线必走这条）
- **Local 必 gitignore**——个人 sandbox 不进 commit（避免泄露个人 prompt 习惯）

#### 2.2 子目录懒加载

- **cwd 之上**：启动即注入（concatenate 链的 part）
- **cwd 之下**（子目录 CLAUDE.md）：**仅在 Claude 读该子目录文件时挂入**——monorepo 噪声控制关键

**实战示例**（monorepo）：

```
monorepo/
├── CLAUDE.md                    # 启动注入：总体工程规约
├── packages/
│   ├── api/CLAUDE.md            # 懒加载：API 设计规约
│   ├── web/CLAUDE.md            # 懒加载：前端组件规约
│   └── db/CLAUDE.md             # 懒加载：DB migration 流程
```

- **效益**：root + 1 个子目录被读 → 只挂入 root + 1 个子目录 CLAUDE.md（避免 4 个全注入撑爆 context）
- **HTML 注释剥离**：`<!-- maintainer notes -->` 这种块级 HTML 注释在注入前被剥掉（code 块内保留）——给 maintainer 留笔记用

#### 2.3 `.claude/rules/` + `paths:` frontmatter

把大 CLAUDE.md 拆成多个小文件，按 glob 触发：

```yaml
# .claude/rules/api-design.md
---
paths: src/api/**/*.ts
---
# API Design Rules
- Use Zod schemas for validation
- All endpoints must have OpenAPI annotations
```

```yaml
# .claude/rules/testing.md
---
paths: **/*.test.ts
---
# Testing Rules
- Use vitest, not jest
- Mock external services
```

- **不加 `paths` 则全量加载**（默认行为）
- **跨文件引用**：`@path/to/import`（递归深度上限 4 跳）
- **AGENTS.md 桥接**：`@AGENTS.md` 或 symlink——和 Aider 等工具的 `AGENTS.md` 共存
- **用户级 rules**：`~/.claude/rules/`（与项目级并行）

#### 2.4 三件套组合（推荐默认配置）

```text
managed policy                  → 锁安全基线（IT 部署）
项目 CLAUDE.md                   → 工程规约（commit 进 git）
.claude/CLAUDE.local.md         → 个人 sandbox（gitignore）
.claude/rules/                  → path-scoped 拆 API/DB/testing 规则
auto memory                     → 沉淀踩坑笔记（Claude 自动）
```

- **效益**：
  - managed 锁死 = 跨员工一致
  - 项目 CLAUDE.md = 团队一致
  - .local = 个人可调
  - rules = 精确触发 + 降 context
  - auto memory = 跨会话学习

### 主题 3：批判性审视 —— 反面分析（5+ 盲点）

#### 盲点 1：Managed policy 不可个人 exclude = 0 信任

**症状**：IT 部署的 managed policy 是「组织意志」——但**任何 developer 都不能 exclude**。如果 IT 写错一条（"永远不要在 commit message 里加 emoji"），团队 100% 必须遵守。

**对照**：GitHub CODEOWNERS 可以 bypass by admin；Kubernetes NetworkPolicy 可以 by namespace override；CC managed policy 0 例外。

**对我们的启示**：教程必须讲清「managed policy 的代价」——IT 必须把它当 production 代码审（code review + 版本化 + 灰度）。

#### 盲点 2：auto memory 25KB 截断 = 知识丢失风险

**症状**：auto memory MEMORY.md 只截前 200 行 / 25KB 注入。Claude 累积的「踩坑笔记」超过这个量后，**老笔记自动从 context 消失**——除非 Claude 自己 `Read` topic 文件。

**对照**：Aider 的 `CONVENTIONS.md` 整段加载；Cursor 的 `.cursor/rules/*.mdc` 按需加载；CC auto memory **有截断阈值**。

**对我们的启示**：教程教「auto memory 维护策略」——定期让 Claude 整理 MEMORY.md（删除过期 / 总结重复），把长 topic 拆到独立文件。

#### 盲点 3：`/compact` 后子目录 CLAUDE.md 不会 re-inject

**症状**：session 跑到 context 满 → `/compact` → **根目录 CLAUDE.md 重新读盘 re-inject**；**子目录 CLAUDE.md 不会自动 re-inject**。如果 Claude 之前依赖某个子目录 CLAUDE.md 的规则，compact 后**这些规则在 context 里消失**。

**对照**：Cursor 的 `.cursorrules` 整段重读；Aider 整段重读；CC 有**子目录不重读**的 bug 行为。

**对我们的启示**：教程必须警告——**关键指令**（如「不要动 secrets」）必须放**根 CLAUDE.md**，不放子目录。

#### 盲点 4：跨机不共享 auto memory = 团队盲区

**症状**：auto memory 存 `~/.claude/projects/<project>/memory/`——**机器本地，不跨机同步**。Developer A 学到的踩坑，Developer B 在自己机器上**看不到**。

**对照**：Cursor 的 `.cursor/rules/*.mdc` 进 git = 全员共享；Aider 的 `CONVENTIONS.md` 进 git = 全员共享；CC auto memory = **个人专属**。

**对我们的启示**：教程「团队记忆」策略——把 auto memory 沉淀的「团队级知识」**人工 commit** 到项目 CLAUDE.md，**只把「个人偏好」留在 auto memory**。

#### 盲点 5：sub-agent memory 与本节 memory 混淆

**症状**：CC sub-agent 有自己的 `memory: user / project / local` 字段（看 `anth-sub-agents` 文档），**与本节 CLAUDE.md / auto memory 是两套并存**——很多用户分不清。

**对照**：CC `Task` tool 派 sub-agent 时，`memory` 字段是 sub-agent **自己的**上下文，不与主 agent 共享。

**对我们的启示**：教程必须**显式区分**——「主 agent 用 CLAUDE.md / auto memory，sub-agent 用自己的 memory 字段」。两者不互通。

#### 盲点 6：Managed policy 与 Project CLAUDE.md 冲突时无明确优先级

**症状**：managed policy 写「不要 npm install」，project CLAUDE.md 写「用 npm install」——官方说 managed 优先，但**没明说如何告警用户**。

**对照**：ESLint config 冲突有明确 warning + exit code；CC managed vs project 冲突**静默**。

**对我们的启示**：教程教「managed policy 调试」——用 `/memory` 看加载清单，发现冲突时人工升级 IT。

### 主题 4：与同类工具的「沿用 vs 错位」决策表

CLAUDE.md / auto memory 不是孤本——Cursor / Aider / Codex 都有类似机制。教程必须讲清「**为什么用 CC 而非 X**」。

| 概念 | CC CLAUDE.md | Cursor `.cursorrules` | Aider `CONVENTIONS.md` | Codex `AGENTS.md` |
|---|---|---|---|---|
| **加载位置** | user message | 全局 | repo root | repo root |
| **跨目录** | 4 级链 + 懒加载 | 整段 | 整段 | 整段 |
| **path-scoped** | `.claude/rules/` + `paths:` | `.cursor/rules/*.mdc` | ❌ | ❌ |
| **自动笔记** | auto memory | ❌ | ❌ | ❌ |
| **企业管控** | managed policy | RBAC | ❌ | approval policy |
| **Git 同步** | ✅（除 auto memory）| ✅ | ✅ | ✅ |
| **教程定位** | 教程「跨会话持久化」主轴 | 教程「对比 Cursor」章引用 | 教程「同类工具」章引用 | 教程「同类工具」章引用 |

**沿用 vs 错位决策**：

| 特性 | 教程决策 | 理由 |
|---|---|---|
| 4 级加载链 | **沿用** | 通用概念（user > project > local）读者已熟 |
| 子目录懒加载 | **沿用 + 加深** | monorepo 必备 |
| `.claude/rules/` + `paths:` | **沿用** | 比 Cursor `.cursorrules` 更细（path-scoped 粒度）|
| auto memory | **沿用** | CC 独有，但概念对标 Aider 的 `.aider.chat.history` |
| managed policy | **沿用** | 企业强需求 |
| HTML 注释剥离 | **错位** | 教程不展开，标「进阶可选」|
| AGENTS.md 桥接 | **错位** | 给 Aider 用户的兼容路径，可选章 |
