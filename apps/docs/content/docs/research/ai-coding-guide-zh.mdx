## Deep-Dive（深读）

### 主题 1：从本源逆向提炼「高质量教程」10 维度评估框架

教程能不能写好，先得知道「好」长什么样。本源是中文圈到目前为止能找到最完整的 Claude Code 教程样本，做对的地方足以反向定义评估维度。下面 10 条每条都有本源的具体证据（file:L），既是夸本源、也是我们的自检表 + 评判其他源的工具。

| # | 维度 | 本源的具体证据 | 为什么算「市场化高质量」信号 |
|---|---|---|---|
| 1 | **章节信息头标准化** | 每篇都有相同 11 字段块（`01-Claude-Code完整安装指南.md:3-14`） | 培训机构 / 高校最看重的可复制性；读者一眼看到学时和难度后能决定要不要点开 |
| 2 | **学习路径分流（≥3 档）** | 固定 4 档「路径 A 快速 / B 完整 / C 排错 / D 专项」（`07-Skills定制完整指南.md:47-108`） | 「不用全看」的承诺降低长教程的阅读阻力；3000+ 行教程不再劝退 |
| 3 | **术语表前置（生活类比）** | 4 列「术语 / 英文全称 / 通俗解释 / 生活类比」，Skill = 手机 APP、MCP = USB 接口（`07-Skills:114-128`） | 把行业黑话翻译为常识；让 0-CS 背景读者能跟上 |
| 4 | **任务驱动而非功能驱动的章节排序** | `00-阅读指南:92`：「不要按功能排，按任务排」；`OC-00:75-90` 的任务 → 文档映射 | 按功能排是文档思维，按任务排是教程思维——教程的价值在引导路径，不在功能罗列 |
| 5 | **版本基线显式化 + 差量追踪** | `适用版本: Claude Code v2.1.169（验证于 2026-06-09）` 标在每篇；CHANGELOG 写差量影响（`CHANGELOG.md:23-33`） | LLM 工具迭代极快，不标版本基线的教程半年内就报废；本源主动管理这个风险 |

| 8 | **作者 voice + 权威溯源** | 每篇开场「老金，我写 X 时...」（`05-Hooks:22`、`07-Skills:23`），同时标 `信息来源: 官方文档 / GitHub` | voice 建立读者信任，权威标注让论断可追溯；两者缺一就变营销文或教材式枯燥 |

### 主题 3：批判性审视 —— 反面分析（盲点 / 视角单一处 / 深度缺失）

本源 39 篇 80 万字看似无所不包，但严肃审视后能列出 7 个具体缺位，每条都是我们教程的差异化机会。

#### 盲点 1：缺「设计哲学」（design rationale）层

**症状**：教 Hook 的事件循环（PreToolUse / PostToolUse / SessionStart…），教 Skill 的目录结构（scripts/templates/），但**很少回答**：

- Anthropic 团队为什么把 Skills 设计成「自动触发」而不是「必须显式调用」？
- Hook 为什么按工具事件分层而不按文件事件分层？
- 为什么 `disallowed-tools` 而不是 `allowed-tools` 是默认配置粒度？

**对照**：claude-skills-blog（Anthropic 官方「How we use skills」blog）整篇都在讲 design rationale。superpowers 的 `writing-skills` 也讲「为什么 SKILL.md 用 frontmatter 而不是 JSON」。本源的 `07-Skills:246-249` 提了「渐进式披露」是核心设计哲学，但只解释了 1 句话「按需加载、避免信息过载」，没有深入「为什么是渐进式而不是全量加载」。

**对我们的启示**：我们的教程每章设一个「为什么这么设计（design rationale）」段，引 Anthropic 团队文章 + 我们自己的工程经验做支撑。这是差异化最容易的一笔。

#### 盲点 2：「评估 / 反馈」机制缺

**症状**：每章开头都说「完成本课后你将能够 X」，但**没有自检题 / 验收清单**让读者知道自己是不是真学到了。`07-Skills:19-35` 列了 11 条「学习目标」，没有 1 道题让读者验证。

**对照**：Coursera / Udacity 等教学平台标配「Quiz / Lab / Project」三件套。本源连 FAQ 都是「错误怎么解」，不是「你知不知道 X」的问。

**对我们的启示**：每章末尾加「自检 3 问 + 实操验收」。例如学完 Hooks 章后：「请用一句话说明 PreToolUse 与 PostToolUse 的设计意图差异；请配置一个能拦截 `rm -rf /` 的 Hook；请说明为什么不应该把 Hook 用作权限边界」。

#### 盲点 3：TDD / 工程方法论这条主线缺席

**症状**：本源教「怎么用工具」，但**几乎不教「用工具的工程方法」**。

- 什么时候让 AI 写、什么时候自己写？没系统讲
- AI 写完代码怎么验？11-企业实战提了 CI/CD 的代码审查，但「我作为开发者下班前怎么知道 AI 改的东西没坏」没讲
- TDD + AI 的工作方式（先写测试再让 AI 写实现）—— 完全没有
- 「外科手术式修改」（`02-基础使用完整指南.md:111-126`）是这块为数不多的好内容，但没有展开成独立章节

**对照**：superpowers 14 个 skill 里 6 个是工程方法论（test-driven-development / systematic-debugging / verification-before-completion / brainstorming / writing-plans / executing-plans）。

**对我们的启示**：我们应该有一整条「与 AI 协作的工程方法论」主线（4-6 章），可借 superpowers 的 14-skill 体系翻译过来做中文版。这条线本源完全没占，是我们最大的差异化空间。

#### 盲点 4：成本意识章缺失

**症状**：术语表提到「Token = AI 处理文字的最小单元，约等于 0.75 个英文单词或 1-2 个汉字」（`快速导航卡.md`），但**没有任何章节系统讲**：

- 一次「让 Claude 重构 100 个文件」大概多少 token / 多少美元？
- `effort = max` 比 `effort = high` 贵多少？什么场景值得？
- 企业月度预算怎么估算？
- 怎么用 `CLAUDE_CODE_MAX_OUTPUT_TOKENS` 等参数控制开销？
- prompt caching 怎么用、能省多少？

LLM 是按 token 计费的工具，缺这一块意味着团队真正用起来就撞预算墙。

**对我们的启示**：教程设独立「成本与预算」章，给出从「我一天 $1 个人开发者」到「我们团队月 $5000」的几档参考画像，配 cost-aware prompt 写法、token 监控配置。

#### 盲点 5：缺失败案例库 + 反例库

**症状**：全部是 happy path —— 「这么配就成功」「这么写就对」。**没有**「我们公司试过把 Claude Code 接入 X 流程，结果失败，复盘原因」这种叙事。FAQ 是解决方案，不是反思。

**对照**：superpowers 的 `test-driven-development` skill 配套 `testing-anti-patterns.md`，列「不该这么写」的反例。

**对我们的启示**：每章配「反例 3 则」+ 整本配「failure mode 库」。失败的素材可以从我们自己的实践来（已有 LEARNINGS.md），更可信。

#### 盲点 6：「我的项目要不要用 AI Coding」决策章缺

**症状**：39 篇都假设读者已经决定用 Claude Code。**但前置问题**「我的项目类型 / 我的团队组成 / 我的合规要求 / 我的代码秘密度 是否适合 AI Coding」没有任何章节。

**对照**：openspec workflow 的第一步就是「proposal」，逼着先问「值不值得做」。

**对我们的启示**：教程第一章不是「安装」，而是「这本教程适不适合你 + Claude Code 适不适合你的项目」决策树。本源把这个反过来——先讲 39 章再让读者自决，门槛更高。

#### 盲点 7：「跟进官方更新」的元方法论缺

**症状**：教程自己说「上游更新很快」「以本机为准」（`README.md:65, 257-259`），但**读者怎么持续跟进**没讲：

- 怎么订阅 anthropic 官方 release？
- 怎么在 v2.1.X 升 v2.1.Y 时识别哪些行为变了？
- 团队怎么建立「AI 工具版本审查」流程？

教程做了「这次升级」，但没教「你以后怎么自己升级」。

**对我们的启示**：教程最后一章是「持续学习与跟进」，把上述元方法论沉淀下来。这是「鱼 vs 渔」差。

### 反例 / 已知坑（自查必看）

- **过度承诺受众**：本源 README 早期写「⭐ 新手能入门 | 开发者能提效 | 团队负责人能建规范 | 企业和高校能做培训」（`README.md:26`），看似覆盖全场景，实际就是「谁都不深」；v4.3 在 CHANGELOG 已「弱化夸张营销词」（`CHANGELOG.md:16`），是作者主动修正的信号。**借鉴**：我们教程一开始就显式声明「这本不教给谁」。
- **营销词残留**：「80 万+ 内容量、1500+ 代码块、250+ 问答」（`README.md:25`）作为数据吸引力强但容易让人误判深度——80 万字摊到 39 章每章 2 万字平均，看似多其实平均；本源仍有「广度 > 深度」的隐性偏好。**借鉴**：我们用「N 个能跑通的项目」「N 条 design rationale」做指标，不用字数 / 数量。
- **过度依赖类比**：生活类比让新手好懂，但「Hook = 汽车传感器」「Channels = 消息总线」用多了会让进阶读者跳出去——他们要的是技术契约，不是隐喻。**借鉴**：进阶章节减少类比、增加规范引用。
- **章节内重复**：每篇都重一遍「术语表 / 路径分流 / FAQ / 排错」，39 章合起来重复内容估计占 15-20%。**借鉴**：把术语表抽到全局；章节内只放该章新术语。
- **「同源多主题」未建模**：本源同时写 CC + Codex + OpenClaw 三工具，但 registry / 索引里没有明确「subjects」字段—— 读者搜索 OpenClaw 时容易错过本源（标签「Claude Code 教程」会让人忽略）。**借鉴**：我们的 registry.json schema 加 `subjects: [...]`（已在 LEARNINGS Section 8.2 Problem 8 沉淀）。
