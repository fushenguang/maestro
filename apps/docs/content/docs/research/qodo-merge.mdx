# qodo-merge（PR-Agent）— Card

## 一句话定位

> 单一垂直 PR 自动化 Agent：把 LLM 包装成 Git 平台原生命令（`/review` `/describe` `/improve` `/ask`），每命令一次 LLM 调用（~30s），跑在 GitHub App / GitLab Webhook / Bitbucket / Azure DevOps / Gitea + CLI + Action + Docker。

## 关键 takeaway

- **vertical agent，不做 generalist**。`pr_agent/agent/pr_agent.py` 用 `command2class` dict 把命令路由到 14 个 `PR*` 工具类；与 CC 的 long-running generalist 形成对比 (methodology:agent/pr_agent.py:L24-L44)。
- **Prompt 即配置**。可调行为落在 `pr_agent/settings/*.toml` + `.pr_agent.toml`（Dynaconf 加载），`apply_repo_settings` 链式覆盖 (methodology:settings/configuration.toml)。
- **多 Provider LLM 抽象**。`pr_agent/algo/ai_handlers/` 提供 `LiteLLMAIHandler / OpenAI / LangChain`，支持 `fallback_models` / `model_weak` / `model_reasoning` + Claude extended-thinking。
- **PR 压缩策略**。`PR Compression` + `Adaptive token-aware patch fitting`：把超大 PR 切成适配 model context 的 patch 序列。
- **多平台部署**。11 个 git_providers + 7 个 servers 形态。
- **SDK 化入口**。`PRAgent(ai_handler=LangChainAIHandler)` 支持注入 handler；GitHub Action YAML 12 行。
- **治理走向**。2025 Qodo 捐给社区，repo 迁到 `The-PR-Agent/pr-agent`，Qodo 转向付费 Qodo 2.0。

## 与 Claude Code 的对比

- 范式：vertical single-task vs generalist long-running。
- 节奏：每工具 1 次（~30s）vs 长时多轮 streaming。
- 部署：App / webhook / Docker / CLI / Action vs 本地 CLI + IDE。
- 配置：`.pr_agent.toml` + Dynaconf vs `settings.json` + skills + CLAUDE.md。
- 多模型：LiteLLM + fallback vs Anthropic 原生 + Bedrock/Vertex。

## 与 Claude Code 教程的关联

- `command2class` 字典模式可类比 CC 的 slash commands / skills registry。

## Brief（梳理）

### 14 工具 `command2class` 路由

`pr_agent/agent/pr_agent.py` `command2class` 字典（`methodology:agent/pr_agent.py:L24-L44`）：

| # | 命令 | 类 | 用途 |
|---|---|---|---|
| 1 | `/review` | `PRReviewer` | 评审 PR |
| 2 | `/describe` | `PRDescription` | 生成 PR 描述 |
| 3 | `/improve` | `PRCodeSuggestions` | 代码改进建议 |
| 4 | `/ask` | `PRQuestions` | 自由问答 |
| 5 | `/update_changelog` | `PRUpdateChangelog` | 自动 changelog |
| 6 | `/add_docs` | `PRAddDocs` | 自动 docstring |
| 7 | `/generate_labels` | `PRGenerateLabels` | 自动标签 |
| 8 | `/help_docs` | `PRHelpDocs` | 文档检索 |
| 9 | `/help_message` | `PRHelpMessage` | 命令帮助 |
| 10 | `/line_questions` | `PRLineQuestions` | 行级问题 |
| 11 | `/similar_issue` | `PRSimilarIssue` | 相似 issue 查找 |
| 12 | `/compliance_check` | `TicketPRComplianceCheck` | 合规检查 |

- **vertical 设计**：每工具 = 1 LLM call（~30s）——失败边界清晰，可独立 retry

### Dynaconf 配置层级（Prompt 即配置）

**3 层配置**（`pr_agent/settings/configuration.toml`）：

| 层级 | 文件 | 用途 |
|---|---|---|
| **defaults** | `pr_agent/settings/configuration.toml` | 全局默认 |
| **global user** | `~/.pr_agent.toml` | 用户级覆盖 |
| **repo level** | `.pr_agent.toml` | 项目级覆盖 |

- **`apply_repo_settings()`** 链式加载：defaults → user → repo（后者覆盖前者）
- **TOML 优先于代码**：prompt 改动不需要改 Python
- **`pr_reviewer_prompts.toml`** 14k 字符——评审 prompt 是产品

### 多 Provider LLM 抽象

`pr_agent/algo/ai_handlers/`：

| Handler | 模型支持 |
|---|---|
| **`LiteLLMAIHandler`** | OpenAI / Anthropic / Bedrock / Vertex / Ollama 等 100+ |
| **`OpenAI`** | OpenAI 原生 |
| **`LangChain`** | LangChain 接入 |

- **fallback_models** 链：主模型失败时自动 fallback 到次级
- **model_weak** / **model_reasoning**：分级使用
- **Claude extended-thinking**：Anthropic 高级推理支持

### PR Compression（自适应切分）

`pr_agent/algo/`：

| 策略 | 用途 |
|---|---|
| **PR Compression** | 把超大 PR 切成适配 model context 的 patch 序列 |
| **Adaptive token-aware patch fitting** | 根据 model 剩余 context 动态调整 patch 大小 |

- **场景**：1000+ 行 PR 用 Opus 4 (200k context) 仍可能溢出——必须切
- **策略选择**：保留 title + description + 关键 diff，省略 boilerplate

### SDK 化入口

```python
from pr_agent import PRAgent
agent = PRAgent(ai_handler=LangChainAIHandler())
result = agent.review(pr_url="https://github.com/.../pull/123")
```

- **Handler 注入**：用户可换 LLM 后端而不动 PR-Agent 逻辑
- **GitHub Action YAML 12 行**：

```yaml
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: qodo-ai/pr-agent@v1
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          OPENAI_KEY: ${{ secrets.OPENAI_KEY }}
```
