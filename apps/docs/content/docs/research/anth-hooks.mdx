# Hooks（官方文档）

## 一句话定位

> Hooks 是 Claude Code 生命周期特定时点自动触发的用户定义处理器（shell 命令、HTTP 端点、MCP 工具、LLM prompt、agent），用于拦截、改写或拒绝 agent 行为。

## 关键 takeaway

1. **事件三层**：会话级（SessionStart / Stop / Notification）、轮次级（UserPromptSubmit / SubagentStart）、工具调用级（PreToolUse / PostToolUse）——全部事件与 JSON schema 见原文 `Hook events` 索引。锚点 `docs:L1`（"Hooks are user-defined shell commands, HTTP endpoints, or LLM prompts..."）。
2. **配置三层**：事件 → matcher group → hook handler；handler 五类 `command` / `http` / `mcp_tool` / `prompt` / `agent`。
3. **执行契约**：command hook 经 stdin 收 JSON——`exit 0` 成功、`exit 2` 阻塞、其它非阻塞；HTTP 用 2xx + JSON body 表达决策；PreToolUse 用 `hookSpecificOutput.permissionDecision`（`allow` / `deny` / `ask` / `defer`）+ `updatedInput`，其它事件用 `decision: "block"` + `reason`。
4. **作用域链**：`~/.claude/settings.json` > `.claude/settings.json` > `.claude/settings.local.json`（gitignore）> 插件 `hooks/hooks.json` > skill / agent frontmatter；企业可用 `allowManagedHooksOnly` 强制只跑托管 hook。锚点 `docs:L1`（"allowManagedHooksOnly to block user, project, and plugin hooks"）。
5. **关键陷阱**：hooks 以系统用户全权限跑——校验输入、`"$VAR"` 引用、阻断 `..`、用 `${CLAUDE_PROJECT_DIR}` 绝对路径、跳过 `.env` / `.git` / 密钥；自 v2.1.139 起 command hook 无 controlling terminal，通知/响铃/改标题必须走 `terminalSequence` 字段（OSC 0/1/2/9/99/777 + BEL）而非直写 `/dev/tty`。锚点 `docs:L1`（"Hooks run without a controlling terminal..."）。

## 与 Claude Code 教程的关联

Hooks 是教程「自动化与治理」章的核心：可用作 lint / format / 注入 secret / 审计日志 / 权限闸口；教程应配套讲清「五种 handler 类型 + 三层事件 + 作用域链 + 安全基线」四件套，避免读者把 hook 写成「能跑但不安全」的形态。

## Brief（梳理）

### Hooks 定位：生命周期时点拦截器

- **核心定位**：Claude Code 生命周期特定时点的「用户定义处理器」
- **5 类 handler 形态**：
  - `command`：shell 命令（stdin 收 JSON）
  - `http`：HTTP 端点（2xx + JSON body 表达决策）
  - `mcp_tool`：调 MCP server
  - `prompt`：用 LLM 做决策
  - `agent`：让独立 subagent 判断
- **与 cron / Makefile 区别**：hooks 是「CC 内部事件驱动」，触发时机严格耦合生命周期

### 事件三层分类

| 层级 | 事件 | 典型用途 |
|---|---|---|
| **会话级** | SessionStart / SessionEnd / Stop / Notification | 注入 context / 清理 / 通知 |
| **轮次级** | UserPromptSubmit / SubagentStart / SubagentStop | 输入验证 / subagent 监管 |
| **工具级** | PreToolUse / PostToolUse / PermissionRequest | 权限闸口 / 输出验证 / 自动重试 |

- 完整事件清单与 JSON schema 见原文 `Hook events` 索引
- **PreToolUse 是「最重」事件**：可做权限决策 + 改写输入（`updatedInput`）

### 执行契约

**command hook**：
- `exit 0` = 成功
- `exit 2` = 阻塞（stderr 进 agent context）
- 其它非 0 = 非阻塞（错误进 stderr）

**HTTP hook**：2xx + JSON body `{decision, reason}`

**PreToolUse 特化字段**：
- `hookSpecificOutput.permissionDecision`：`allow` / `deny` / `ask` / `defer`
- `hookSpecificOutput.updatedInput`：改写 tool 输入

### 作用域链（5 层优先级）

```
~/.claude/settings.json     → user 级（个人跨项目）
.claude/settings.json       → project 级（commit 进 git）
.claude/settings.local.json → project 本地（gitignore）
插件 hooks/hooks.json        → plugin 级
skill / agent frontmatter   → skill/agent 内联
```

- **企业管控**：`allowManagedHooksOnly` 强制只跑托管 hook，**完全屏蔽 user/project/plugin**——给企业「员工带自己 hook 进公司仓库」的 0 信任能力

### 关键陷阱（8 条安全基线）

| 陷阱 | 解决 |
|---|---|
| Hook 以**系统用户全权限**跑 | 输入校验 + 权限收敛 |
| 字符串拼接未加引号 → 命令注入 | 永远 `"$VAR"` |
| 未阻断 `..` 路径 → 跨目录写 | 校验路径前缀 |
| 路径相对 → 不同 cwd 行为不同 | 用 `${CLAUDE_PROJECT_DIR}` 绝对路径 |
| 直读 `.env` / `.git` / 密钥 | 显式排除 |
| v2.1.139+ 无 controlling terminal → 通知失败 | 用 `terminalSequence` 字段（OSC 0/1/2/9/99/777 + BEL） |
