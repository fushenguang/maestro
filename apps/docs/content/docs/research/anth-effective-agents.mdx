# Building Effective Agents（Anthropic Engineering）

## 一句话定位

> 成功不在于做最复杂的系统，而在于做最匹配需求的系统：先简单 prompt + 评测，多步 agent 仅在简单方案不够用时再上。

## 关键 takeaway

1. **Workflow vs Agent 的架构性区分**：workflow = 「LLM 与 tools 通过预定义代码路径编排」；agent = 「LLM 自主决定步骤与工具调用」。锚点 `blog:L5, L6`。
2. **「先用最简」原则**：agentic 系统用延迟与成本换性能，多数场景下 retrieval + in-context examples 优化的单次 LLM 调用就够。
3. **五大基础模式**（augmented LLM → workflow → agent 渐进谱系）：
   - **Augmented LLM**：building block = LLM + retrieval / tools / memory + MCP。
   - **Prompt chaining**：任务分解为固定子任务序列，可加 gate（programmatic check）做中间校验。
   - **Routing**：按输入分类路由到不同下游 prompt / tool；典型应用是「简单问题用 Haiku、难问题用 Sonnet」的 cost / perf 优化。
   - **Parallelization**：两种变体——**Sectioning**（独立子任务并行）+ **Voting**（同任务多 prompt 多票），用于 guardrail / code vuln 评审 / inappropriateness 多维判定。
   - **Orchestrator-workers**：中央 LLM 动态拆任务、派 worker、合成结果；与 parallelization 区别是「subtasks aren't pre-defined, but determined by the orchestrator based on the specific input」，适合编码类「不知道要改几个文件」的任务。锚点 `blog:L54`。
   - **Evaluator-optimizer**：一个 LLM 生成、另一个评估反馈、循环迭代；fit 信号：「LLM 响应能在人类反馈下显著改进 + LLM 自己能给出这种反馈」。锚点 `blog:L61`。
4. **Agent 的触发条件**：开放式问题且步数无法预测、不能硬编码固定路径；并需在 sandbox 充分测试 + 加 guardrail。
5. **三条核心原则 + 工具是 ACI**：（1）保持设计简单；（2）显式展示 agent 的规划步骤；（3）通过详尽的工具文档与测试精心打磨 agent-computer interface (ACI)——SWE-bench 上「优化工具的时间比优化 prompt 还多」。锚点 `blog:L83, L86`。

## Brief（梳理）

### Workflow vs Agent：架构性区分

- **Workflow**：LLM 与 tools 通过**预定义代码路径**编排
- **Agent**：LLM **自主决定**步骤与工具调用
- **核心选型原则**：成功不在于做最复杂的系统，而在于做**最匹配需求**的系统——先简单 prompt + 评测，多步 agent 仅在简单方案不够用时再上

### 「先用最简」原则

- agentic 系统用**延迟与成本换性能**
- 多数场景下：`retrieval + in-context examples` 优化的**单次 LLM 调用**就够
- 反例：上来就 multi-agent 框架 → 90% 场景过度设计

### 五大基础模式（augmented LLM → workflow → agent 渐进谱系）

| 模式 | 抽象度 | 典型场景 |
|---|---|---|
| **Augmented LLM** | building block | LLM + retrieval / tools / memory + MCP |
| **Prompt chaining** | workflow | 任务分解为固定子任务序列；可加 gate（programmatic check）做中间校验 |
| **Routing** | workflow | 按输入分类路由到不同下游 prompt / tool；典型「简单问题 Haiku / 难问题 Sonnet」cost 优化 |
| **Parallelization** | workflow | 两种变体：**Sectioning**（独立子任务并行）+ **Voting**（同任务多 prompt 多票）|
| **Orchestrator-workers** | workflow → agent | 中央 LLM 动态拆任务、派 worker、合成结果；**subtasks aren't pre-defined**（编码类「不知道要改几个文件」天然适用）|
| **Evaluator-optimizer** | workflow | 一个 LLM 生成、另一个评估反馈、循环迭代 |

### 两种 Parallelization 详解

- **Sectioning**：独立子任务并行（每 worker 跑不同子任务，**结果合流**）
  - 例：guardrail 评审 + code vuln 评审 + inappropriateness 多维判定
- **Voting**：同任务多 prompt 多票（每 worker 跑**同一任务**的不同版本，**多数决/取优**）
  - 例：温度 0.7 跑 5 次取最常见答案（事实型 QA）

### Agent 触发条件

- **开放性问题**且**步数无法预测**、**不能硬编码固定路径**
- **必要前置**：
  - 在 sandbox 充分测试
  - 加 guardrail
  - 控制单次 LLM 调用成本上限

### 三条核心原则 + 工具是 ACI

1. **保持设计简单**——别为「未来可能需要」做复杂
2. **显式展示 agent 的规划步骤**——prompt 里让 agent 显式输出「我现在要做什么 / 为什么 / 下一步」
3. **ACI（Agent-Computer Interface）**：把 tool 当 API 写——**详尽 docstring + 错误处理 + 绝对路径 + poka-yoke（防呆）**
- **SWE-bench 实测**：「优化工具的时间比优化 prompt 还多」——ACI 是 LLM-as-agent 的真正瓶颈
