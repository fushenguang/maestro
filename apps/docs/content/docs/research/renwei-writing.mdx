# 人味儿写作 · Renwei Writing

## Card（速览）

### 一句话定位

> 一个诞生于凌晨五点失败的中文写作 skill：当 AI 去改一个人的文字时，保证改完之后那个人还在。核心是"少动、保手迹、隐形"的改稿哲学，配以中文适配的 AI 痕迹事后检查清单。

### 关键 takeaway

1. **人味儿 = 位置 + 代价 + 手迹**。位置：写作者站在具体的地方说话（凌晨五点的那个位置决定了他写"拯救自己的前额叶"而不是"提升专注力"）。代价：真句子用注意力换来的，读者闻得出哪句背后有生活、哪句只有词库。手迹：那些多余的"呢""吧"、忽长忽短的呼吸——每个人独有的说话样子。(SKILL.md:L29-L42, README.md:L19-L28)

2. **"越磨越用力"是默认失败路径**。原文：AI 磨三遍，一遍比一遍漂亮——对仗更工整、用词更讲究、加金句。结果：作者说"人味儿变少了"。最痛的一处：原文"我们要拿什么来拯救自己的前额叶呢？"，AI 精简为"我们拿什么拯救它？"——句子干净了，叹气的"自己的"和"呢"被删了，叹气的人不在了。(SKILL.md:L18-L26, README.md:L5-L17)

3. **六条操作规则**，按优先级：(1) 只做减法，一段动三处是常态，动十处是事故；(2) 毛边先当手迹不当瑕疵——删之前问"说话的人还在吗"；(3) 改稿时写出漂亮句子是警报——那是你在表演，不是他在说话；(4) 改完跑事后检查清单（只查动过的句子）；(5) 逐处交代改动理由，拿不准的标出来；(6) 上限是隐形——不追求"改得更好"，追求"清掉绊脚的，其他原样"。(SKILL.md:L44-L57)

4. **两层结构，顺序不可逆**：底层原理管"改之前"——这句该不该动？事后清单管"改之后"——动过的地方有没有引入 AI 味道？顺序反了（先跑清单再想动不动），就会拿清单去扫作者的原文，把手迹当瑕疵修。(SKILL.md:L78-L93, README.md:L51)

5. **与 humanizer 的关系**：renwei 的 post-edit-checklist 直接提炼自 blader/humanizer（MIT），针对中文写作做了适配。humanizer 提供技术骨架（33 种模式），renwei 提供哲学框架（少动、保手迹）。两者互补形成"英文引擎 + 中文适配 + 改稿哲学"的完整体系。(README.md:L49, post-edit-checklist.md:L1-L5)

### 与去 AI 味写作的关联

- 这是我们要建的 /de-ai skill 的哲学底座。renwei 解决的是"为什么不能改太多"（人味儿的定义），humanizer 解决的是"改什么"（33 种模式）。两者合起来才能同时回答"动不动"和"怎么动"。
- renwei 最值得移植的是 gotchas 机制——每次失败换一条 gotcha，写入 skill 防止再犯。我们要在自己的写作实践中持续积累"教程写作专用 gotchas"。

## Brief（梳理）

### 仓库结构

纯 Markdown skill，3 个核心文件：(1) `SKILL.md`（~250 行）：心法本体——人味儿的定义（位置/代价/手迹三要素）、六条操作规则、四条 gotchas（真实失败换来的）、两层结构说明；(2) `references/case-study.md`（~80 行）：催生本 skill 的真实失败案例完整解剖——原文、改砸的版本、被接受的版本，三版并排，逐条分析失败点；(3) `references/post-edit-checklist.md`（~110 行）：事后检查清单，提炼自 humanizer+Wikipedia，按中文写作做了六组适配。

### 人味儿三要素

**位置**：AI"哪儿都能站，所以哪儿都不在"。凌晨五点、看着朋友被消息提醒拽走的那个位置，决定了一个人写"拯救自己的前额叶"而不是"提升专注力"。位置的不可替代性就是存在感。(SKILL.md:L30-L33)

**代价**：真实的句子背后有真实的观察、被折磨过的经历。每个判断都是付过注意力换来的。AI 写漂亮话不付任何代价——读者隔着屏幕能闻出来：哪句话背后有生活，哪句话背后只有词库。(SKILL.md:L34-L36)

**手迹**：两个人抄同一段话，内容一样，你仍认得出谁抄的。看似冗余的"自己的"、句尾的"呢"、忽长忽短的呼吸——都是手迹。AI 没有手，怎么写都像打印的。(SKILL.md:L37-L42)

### 失败案例解剖（case-study.md 精要）

**原文（橘子）**：一段关于 ADHD 和注意力分散的观察笔记，口语化、有观察者的冷静、结尾带叹气的"呢"。

**AI 磨三遍的失败版本**："小红点""奢侈""稀有""刚集中被叫走刚散掉又得集中"——越来越"漂亮"但六处要害：(1) "细小琐碎的事情"→"小红点"：换成了 AI 的画面；(2) "变得越来越不可能"→"已经是一种奢侈"："奢侈"是词库里的漂亮词；(3) "进入心流也越来越难"→"心流更是稀有"：丢了口语感；(4) "注意力集中和注意力涣散的循环"→对仗句：把观察者冷静的陈述改成了表演——原句是全文最好的句子；(5) 捏造"几百次"的数据；(6) "拯救自己的前额叶呢？"→"拯救它？"：删掉叹气和自嘲。

**被接受的版本**：只动三处——删一个重复的词、加逗号留呼吸、逗号改句号。其余全部保留。(case-study.md:L1-L81)

### 六条操作规则（完整）

1. **只做减法**：每一处改动都要能说出"原文这里真正打绊了"，说不出来就不改。一段动三处是常态，动十处是事故。
2. **毛边先假设是手迹**：冗余、口语、不工整的对仗、啰嗦的语气词，先假设是这个人说话的样子。删之前问：删掉以后，说话的人还在吗？
3. **不写金句**：改稿时写出工整的对仗、漂亮的比喻、"奢侈""稀有"级别的词——这是警报。那是你在表演，不是他在说话。
4. **改完跑事后检查**：只扫动过的句子，不扫原文。六组检查：意义拔高、宣传腔、句式套路、格式痕迹、语气痕迹、填充与对冲。
5. **逐处交代改动**：列出每一处动了什么、为什么。拿不准的明确说"这处拿不准，可以还原"。否决权永远在作者手里。
6. **上限是隐形**：最好的结果是读者完全感觉不到你来过。不要追求"改得更好"，追求"清掉绊脚的，其他原样"。

### 四条 Gotchas

1. **越磨越用力是默认失败**：用户说"打磨三遍"不等于三遍逐级加工。第一遍理顺后，后两遍最常见正确动作是把第一遍改过头的地方还原。
2. **平直陈述句不要动**：如"是 A 和 B 的循环"——平铺直叙本身就是观察者的冷静，改成对仗是把人话改成文案腔。
3. **删语气词等于删人**："呢""吧""了"这些字在书面语标准下是冗余，但携带叹气、自嘲、犹豫。
4. **抽象换具体要克制**："细小琐碎的事情"换成具体画面——那是你的画面，不是他的。除非原词真的含糊到读者会卡住。

## Deep-Dive（深读）

### 主题 1：renwei 与 humanizer 的哲学层对比——为什么两个都需要

**核心张力**：humanizer 是"修掉所有 AI 痕迹"（追求干净），renwei 是"改完之后那个人还在"（追求保留存在感）。单独用 humanizer → 可能过度编辑，把所有不完美的粗糙处都磨平——结果 AI 味没了，人味也没了。单独用 renwei → 缺乏系统化的"AI 味道检测清单"，凭感觉判断，容易漏掉真正的 AI 痕迹。

**解决**：renwei 的"少动"原则作为门禁，humanizer 的 33 种模式作为工具箱。流程是：先判断动不动（renwei 哲学）→ 如果动，用模式识别改什么（humanizer 技术）→ 改完跑清单（renwei 的检查清单 = humanizer 的中文适配版）。

**关键区别**：

| 维度 | humanizer | renwei |
|---|---|---|
| 底层信念 | AI 文本有明确的、可枚举的模式（33 种） | "人味"不可被还原为模式，改得越多丢得越多 |
| 对"干净"的态度 | 干净 = 好 | 干净 = 警惕（"打印体"） |
| 对"毛边"的态度 | 系统化误伤防护（12 种不算 AI 的情况） | 毛边先当手迹，删前问"人还在吗" |
| 操作量期望 | 全面改写（draft → audit → final） | 一段动三处是常态（减法优先） |
| 交付物 | 完整的改写结果 | 改动的清单 + 理由 + 拿不准的标记 |
| 适用范围 | 通用文本（从 Wikipedia 条目提取的规律） | 个人化文本（帮人改稿、代笔返工） |

### 主题 2：post-edit-checklist 与 humanizer 33 模式的对应关系

renwei 的清单将 humanizer 的 33 种模式压缩为六组，并做了中文适配：

| renwei 清单组 | 对应 humanizer 模式 | 中文适配变化 |
|---|---|---|
| 一、意义拔高 | #1 意义拔高 + #3 浅层分析 | 增加"标志着""见证了""体现了"等中文高频拔高词 |
| 二、宣传腔 | #4 宣传腔 | 中文黑名单：璀璨、深厚底蕴、赋能、打造、致力于、匠心、不容错过、极致体验 |
| 三、句式套路 | #9 否定排比 + #10 三连 + #11 同义词轮换 + #12 假范围 + #31-33 punchline/格言/假坦诚 | 新增中文特有：自问自答（"结果呢？惨烈。"）、短句轰炸、"重要的不是…而是…" |
| 四、格式痕迹 | #14 破折号 + #15 加粗 + #18 emoji + #29 碎片标题 | 破折号从英文"硬删"降级为中文"逐个替换"（承认中文写作破折号更常见） |
| 五、语气痕迹 | #20 聊天残留 + #21 知识截止 + #22 谄媚 + #5 模糊归因 + #25 万能结尾 + #28 签到预告 + #6 公式化挑战段 | 中文版：签约式过渡"让我们深入探讨"、万能展望"未来可期"、挑战与展望段 |
| 六、填充与对冲 | #23 填充 + #24 过度对冲 | 中文版：值得注意的是、不得不说、在某种程度上或许可能 |

renwei 清单保留了 humanizer 的"误伤防护"和"人写迹象"两节，但做了关键升级：(1) 明确"只扫动过的句子"——这是 renwei 独有的操作规范，humanizer 没有；(2) "人写迹象"节与 SKILL.md 的"手迹"概念形成交汇——同一件事的两面，确保改稿时保留、检查时不误修。

### 主题 3：renwei 的 gotchas 机制对 Skill 设计的启发

renwei 的四条 gotchas 不是从理论推出来的，是从"凌晨五点改砸了"的真实失败中换来的。这种"每次失败→沉淀一条 gotcha→写进 skill 防止再犯"的机制，是 skill 持续进化的核心引擎。

对我们建立 /de-ai skill 的启发：

1. **Skill 需要 gotchas 区**：不是"你应该怎么做"的正面指令，而是"我们上次XX情况翻车了，别再来一次"的血泪教训。正面指令容易被忽略，翻车故事有场景、有情绪、记得住。

2. **Gotcha 格式应该保留原事件的冲击力**："删语气词等于删人"比"注意保留语气词"有力一百倍。保留碎片的对话感——"句子干净了，人没了"——比客观描述更有约束力。

3. **教程写作专用 gotchas 候选**（预判，待实践验证）：
   - "技术教程也有手迹"：同一个概念，A 作者用"把…拽到…"、B 作者用"将…迁移至…"。不要统一。
   - "代码注释不要去 AI 味"：注释是给 AI 和人类一起看的，过度润色注释反而降低可读性。
   - "步骤编号的节奏是手迹"：有人写 1-2-3，有人写"首先→然后→最后"。保留作者的步进节奏。

### 主题 4：两个源合成 /de-ai skill 的完整方案

**架构**（已在 humanizer Deep-Dive 主题 3 中详述）：
- Layer 1（哲学层）：来自 renwei——少动、守手迹、隐形是上限
- Layer 2（检测层）：来自 humanizer——~28 种中文化模式 + 中文特有补充
- Layer 3（审计层）：融合两者——改完跑清单（renwei），只查动过的句子（renwei），残留 AI 味标出（humanizer 审计机制）

**关键创新点**（两个源都没有但我们应该加）：
1. **声纹库**：用团队已有的教程文章作为声纹样本，使改写不偏离团队风格
2. **激进程度分级**：轻量去味（删聊天残留、emoji、签到语）vs 深度重写（改句式、换高频词），用户可选
3. **教程场景专项优化**：教程写作有自己独特的 AI 味模式（签到过渡、假坦诚开场、排比三连、万能结尾、碎片标题、谄媚语气——这 6 种是教程高频模式）
4. **持续 gotchas**：每次翻车→加一条 gotcha→进 skill

### 与教程写作的关联

- renwei 的"只做减法"直接绑定到教程写作的改写场景：教程修改的目的是"清掉绊脚石"（跳步、歧义、术语不一致），而不是"让文采更好"。文采是附带的，准确是首要的。
- case-study.md 可作为 skill 的"为什么不能改太多"的叙事锚点——比抽象原则有说服力得多。
- renwei 开头的故事（凌晨五点、改砸了三遍、用户说人味儿变少了）本身就是一篇高"人味儿"的 skill 写作示范——我们的 skill 也应该用具体的失败场景开场，而不是抽象定义。
