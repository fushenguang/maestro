sunny 亲自按教程操作后的 4 个核心发现：

00 篇的"为什么大多数教程你读不完"整段是教育哲学元分析——读者打开教程地图是想找入口，不是来听教育行业批判的。虚构第二人称（"你花了三个晚上..."）是 AI 味的头号来源。

一套内容无法同时服务"npm 是什么"的读者和"教我怎么配 MCP"的读者。受众没定 = 写作视角飘忽 = 内容怎么写都不对。

- **01 章**：安装体验写得太简单——Mac M1 + brew 安装的多次报错和代理配置没体现。推荐使用 cc-switch（https://github.com/farion1231/cc-switch）简化模型代理配置。

## 核心决策

### 决策 1：锁定主受众

**主受众：会写代码、会用终端、但不熟悉 AI 编程工具的技术人。**

对应调研标签：`CS=yes / DevOps=mid / CC=newbie`。

具体画像：
- 知道 `npm` 是什么、日常用 git
- 写过项目（前端或后端均可）
- 但从未让 AI 帮他写过代码
- 对 CC 能做什么、怎么用、有什么坑——完全不知道

**为什么不选"完全零基础"？**
Claude Code 是开发者工具。教不会终端的人用 CC，教程会变得冗长，且真正需要这个教程的开发者会被劝退。诚实声明前置条件 > 假装能零基础教会。

**为什么不完全放弃其他受众？**
路径 B（有具体问题的 CC 用户）和路径 D（团队负责人）仍然可以通过 flag 分流——在 00 地图中标注"路 D 读者：直接跳到第 20 章"。但内容的主体视角锁定主受众。

### 决策 2：建立调研绑定流程

后续每章写作必须遵守：

1. **写前**：从 `research/registry.json` 列出本章关联的调研源
2. **写中**：每个关键概念至少引用 1 个调研源的具体洞察（含锚点）
3. **写后**：在 frontmatter 的 `research_sources` 中列出实际使用的源

这解决了"调研和写作两张皮"的问题。

### 决策 3：每章 frontmatter 新增三个字段

```yaml
---
audience: "🟡 需要编程基础"  # 🟢🟡🔴 三级
cc_version: "2.1.174"
research_sources:
  - anth-claude-code-overview
  - anth-slash-commands
---
```

受众 flag 体系：
- 🟢 = 零基础可读（目前只有 00 导航章部分内容）
- 🟡 = 需要有编程基础（大部分项目章和功能章）
- 🔴 = 需要特定技术栈（Tauri 章需 Rust、Bevy 章需 ECS）

### 决策 4：04 章去掉 Supabase

排行榜改用纯前端方案（localStorage），或进程内内存。不引入新概念。自然引出 Memory（跨 session 记住规范）+ Hooks（自动检查）——不需要一个外部数据库来做这件事。

### 决策 5：重写范围

全部 5 篇（00-04）重写。不是修补——是按住受众视角重新讲一遍。

## AI 味的根因分析

AI 味不是"写得不好"的问题——是**受众不确定时的系统行为**：

1. 当模型不知道读者是谁 → 默认策略是"对所有人说话"
2. "对所有人说话" → 泛化的修辞、哲学式开场、虚构的第二人称
3. 泛化的内容 → 没有真实细节（因为不知道读者的真实痛点是什么）
4. 没有真实细节 → 读者感觉"看了没感觉，看不下去"

**修复不是"降低 AI 参与度"——是给 AI 一个清晰的读者画像。** 当模型知道"我在跟一个会写 Vue 但不信任 AI 的前端开发说话"，它的输出会自动变得更具体、更少泛化修辞。

这验证了之前讨论中的核心洞察：**教程的质量公式 = 娱乐钩子 × 认知摩擦力 / 思考替代量。** 没有明确受众，就无法设计认知摩擦力——你不知道读者哪里会觉得难，哪里会觉得简单。

## 元笔记：教程写作 Skill 的候选要素

本轮迭代中浮现的可复用方法论（最终会整理成 SKILL.md）：

1. **受众锁定**：写任何教程前，先定义 `CS背景 × DevOps熟练度 × 工具经验` 三轴受众画像
2. **调研先行**：所有关键概念必须有调研源支撑（Card/Brief/Deep-Dive），不凭空写
3. **真实经历锚点**：每章至少嵌入 1 个真实踩坑经历——这是 AI 无法凭空编造的内容
4. **事实验证**：所有命令、API、版本号必须在实际环境中验证（不能靠记忆或文档推测）
5. **AI 味检测**：出现以下特征 → 重写：(a) 虚构第二人称场景 (b) 超过 3 句的哲学开场 (c) "为什么大多数 X 都 Y" 句式 (d) 没有具体细节的泛化修辞
6. **受众 flag**：每章标注 🟢🟡🔴 三级受众标签
7. **版本声明**：工具类教程必须声明编写时使用的工具版本
