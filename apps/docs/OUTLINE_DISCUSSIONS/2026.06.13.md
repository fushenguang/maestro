## 讨论起点

上次 session（2026.06.12）完成了 4 轮深度讨论 + 第一次收敛，核心决策：

1. **教程的第一性不是"教"，是"拉"** —— 每章开头不让读者觉得"我应该学这个"，而是"靠，这就是我昨天遇到的问题"
2. **从 feature-centric 转为 use-centric** —— 不是"CC 有哪些功能"，是"用户要做什么 → 做的时候遇到什么问题 → CC 的什么功能解决"
3. **多个独立小项目 + 几个大项目** —— 4 类（游戏/网站/商业/学习），TypeScript/Rust
4. **纯功能章保留** —— 理论和实践相互成就，做项目时遇到的功能回头深度理解
5. **每章必须有混子哥级别的开场场景（3 句话好故事）** —— 找不到的章不配存在
6. **教程 = 说服，不是教学** —— ethos + pathos + logos 结构
7. **情绪通路 > 意志力通路** —— 让知识走情绪通路，绕过意志力的收费站
8. **高质量教程公式**：娱乐钩子 × 认知摩擦力 / 思考替代量

### Step 2：确定章序

核心原则：
1. **先做出东西，再回头理解** — 项目章在功能章之前（03-04 → 05-09）
2. **先 Web 后桌面** — Tauri 章节在读者理解"做什么"之后（03-04 Web → 05-09 功能 → 10-12 Tauri）
3. **先免费后付费** — 商业变现放在读者已经有作品之后（13 在 10-12 之后）
4. **情感弧线上升** — 好玩（03-04）→ 有用（05-12）→ 赚钱（13）→ 给家人用（14）→ 专业（15-16）→ 生态参与（17-20）

章序决策中讨论过的备选方案：

**备选 A：商业变现（13）提前到 Tauri 之前**
- 理由：赚钱是最高情绪价值，早点出现早点兴奋
- 否决原因：Stripe + MCP 复杂度需要读者先理解 MCP（08）和 Sub-agents（09），且 Newsletter 不是"做出来就赚钱"——需要积累读者。放在 Tauri 实践之后，读者更有耐心处理 Stripe 审核等真实商业摩擦

**备选 B：学习工具（14）提前到篇一**
- 理由：做给孩子的工具是最强的情感钩子
- 否决原因：Anki（SM-2 算法）+ 数学闯关（游戏化设计）需要的 Skills/Memory/Hooks 深度在 03-04 还没讲——读者会"照做但不理解"。放在篇四，读者可以自己改题库、改算法、改 UI——真正为自己孩子定制

**备选 C：Bevy（17）放在 09（Sub-agents）之后**
- 理由：Sub-agents 章用的 Phaser（TS），Bevy（Rust）是更高级的并行
- 否决原因：Bevy 需要 Rust 基础（macroquad 太空射击的场景铺垫）+ ECS 思维转换需要读者有足够的编程经验积累。放在篇六，作为"你已经会了很多，来挑战最难的东西"——是奖励，不是门槛

### Step 3：功能章的顺序和场景设计

核心机制（05-09）的顺序：Hooks → Memory → Skill → MCP → Sub-agents

这个顺序不是随机的——是一个"问题升级链"：

```
你做完了贪吃蛇（03-04）
    ↓
游戏上线了——CC 乱动你的数据库 → Hooks（05）是安全带
    ↓
你每次重开 session 都要重讲规则 → Memory（06）是长期记忆
    ↓
你发现每次都口头教太累了 → Skill（07）是可复用资产
    ↓
你想接外部 API 但 CC 访问不了 → MCP（08）是外部感官
    ↓
你做更复杂的游戏 一个人忙不过来 → Sub-agents（09）是团队
```

每一章的场景都"长"在上一章的痛点之上——不是独立的功能介绍，是一个连续的问题解决链。

**但每章仍然是独立小项目场景**——读者可以跳章读。只是连着读的人会感受到"问题自然升级"的叙事弧线。

### Step 4：Tauri 三章的内部关系

10 → 11 → 12 是一个"复杂度递进 + 场景拓展"链：

```
10. Tauri 入门（贪吃蛇）
  - 读者已经会做 Web 贪吃蛇（03-04）
  - 这章教：怎么把它包装成桌面应用
  - Tauri 核心概念：command / invoke / capabilities / 窗口管理
  - 数据：SQLite 本地存储（替代 Supabase）
  - 读者学会：Tauri 的基本骨架

11. Tauri Markdown 笔记
  - 全新的桌面项目（不是迁移）
  - Tauri 中级概念：文件系统监听 / FTS5 全文搜索 / 三栏布局
  - 读者学会：Tauri + 原生 OS 功能（文件系统 + 通知 + 托盘）

12. Tauri Git 客户端
  - 最复杂的桌面项目
  - Tauri 高级概念：原生 C 库绑定（git2-rs）+ Monaco diff + SSH 密钥管理
  - 读者学会：Tauri + 第三方原生库 + 安全存储（OS keyring）
```

如果读者只做一章——第 10 章足以让他们理解 Tauri 并能自己做桌面工具。11 和 12 是"如果你想要更酷的东西"。

### Step 6：每章的"混子哥级 3 句话场景"设计

3. **第三句**：承诺——这一章教你解决它（"这一章..."）

## v2 大纲的关键设计决策

### 决策 1：为什么贪吃蛇是第一个项目？

备选：Markdown→微信转换器、Anki 卡、数学闯关

选择贪吃蛇的理由：
- **即时满足最强**：写完就能玩——比"工具"更快出情绪反馈
- **视觉反馈明确**：蛇在屏幕上移动 vs 转换器"复制出来格式对"——前者更直观
- **扩展路径自然**：加排行榜（Memory + Hooks 初体验）、加桌面版（Tauri 入门）——一个项目串起 3 章
- **受众最广**：所有人都玩过贪吃蛇——不需要解释"这是什么"

### 决策 2：为什么 Tauri 三章放在中间（10-12）而不是篇一或篇六？

- 不在篇一：Tauri 需要 Rust + 前端 + 桌面配置，零基础读者装都装不上
- 不在篇六：Tauri 是本教程核心差异化——太靠后读者还没看到就走了
- 在篇二之后（05-09 后）：读者已经理解了 CC 的核心功能，Tauri 章是"把这些功能用在一个更复杂的项目里"——是应用，不是学习

### 决策 3：功能章（05-09）为什么不用独立项目而用"场景"？

Hooks 深度（05）用贪吃蛇的场景、Memory（06）用多项目场景、MCP（08）用贪吃蛇排行榜需求——这些不是独立新项目。

理由：功能章的目的是**深度理解**，不是**做出新东西**。如果每章都要做一个新项目，读者会累死。场景的作用是"让读者进入问题情境"——一旦进入，讲解可以更系统。

### 决策 4：为什么工程方法论（15）放在项目实战之后？

传统教程把"方法论"放在前面——先讲流程再实战。v2 倒过来：

理由：读者先无流程地做了 8+ 个项目，撞上了"快但乱"的痛——然后 15 章说"你之前的痛苦有个名字——叫'没走流程'"。这时候读者不是在"学方法论"——是在"找解药"。

### 决策 5：Bevy（17）作为 Rust 进阶压轴，不是 Rust 入门

Bevy 不是给"想学 Rust"的读者的——是给"已经有点 Rust 基础、想要一个硬核挑战"的读者的。

如果读者没有 Rust 基础：06 章（Memory）提了 Rust CLI 场景、09 章（Sub-agents）提了 Rust 项目的 sub-agent 架构——这两处提供 Rust 的"侧面接触"。如果读者想专门学 Rust——本教程不代替 Rust 入门教程。但 Bevy 章告诉读者："你学了 Rust 之后，能做出什么 TS 做不了的东西"。

## 开放问题 & 待讨论

1. **第 16 章（Git + Prompt + Settings）三合一是否需要拆分？** 三个主题共享"工程素养"框架，但每个都够独立成章。当前三合一——实战写作时如果每个主题超过 1500 字，就拆成两章（15 工程方法论 + Git 工作流 / 16 Prompt 工程 + 项目配置）。
