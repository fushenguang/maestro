## 篇〇：入口（1 章）

### 00. 教程地图：怎么读这教程

- **类型**：导航章
- **3 句话场景**：「你花了三个晚上看完教程。发现自己需要的章节全在最后三章——而前三章跟你没什么关系。这本教程不这样。」
- **是什么**：4 条阅读路径分流——"我想先玩起来" / "我有个具体问题要解决" / "我想系统掌握 CC" / "我是团队负责人要评估"。每条路径指向不同的章序。
- **为什么**：教程不是小说，没人从头读到尾。路径分流的本质是降低决策成本——让读者 5 分钟内找到自己的入口。

## 篇一：从零到第一个作品（4 章）

> 情绪锚点：**先玩起来**。读者在 2 章内装好 CC，4 章内做出一个能发给朋友的网页游戏。

### 01. 安装配置：翻墙、借卡、换号——三天了还没装上

- **类型**：环境章
- **项目**：无（纯环境准备）
- **语言**：—
- **复杂度**：—
- **3 句话场景**：「你想装 Claude Code。官网说要 Anthropic 账号。你没有。你找朋友借了个号——被封了。你发现中国用户装个 CLI 工具，得先翻三座山。这一章带你一座一座翻过去。」
- **是什么**：装好 CC + 跑通第一个 prompt。覆盖中国市场独特障碍——VPN、支付、封号、API key 获取。
- **为什么**：如果装不上，后面所有内容作废。先解决"能不能用"，再谈"怎么用好"。
- **CC 功能首次出现**：安装、`claude` 命令、第一个 prompt

### 02. 对话基础：你让 CC 修 bug，它帮你重写了整个项目

- **类型**：交互章
- **项目**：一个现成的 buggy TypeScript 小项目（故意埋 bug）
- **语言**：TypeScript
- **复杂度**：简单
- **3 句话场景**：「你有一个 TypeScript 项目，一个函数老是返回 undefined。你让 CC 修这个 bug。它看了一下——然后开始重写你的整个路由层。你吓得按了 Ctrl+C。这一章教你跟 CC 说话的边界。」
- **是什么**：掌握 slash commands + 理解权限弹窗三种选择（allow / deny / always allow）什么时候用。学会控制 CC 的"热情"——告诉它什么时候动手、什么时候只动嘴。
- **为什么**：CC 是对话驱动工具。不知道什么时候该点 allow，用户要么全点 yes（不安全），要么全点 no（用不了）。两者都导致弃用。
- **CC 功能首次出现**：`/prompt`、权限系统、`/clear`、`/compact`

### 03. 贪吃蛇 + 全球排行榜（上）：Skills——1 小时做出能玩的网页游戏

- **类型**：项目章（小游戏）
- **标签**：🎮 游戏
- **项目**：Snake + 排行榜 — TypeScript + Canvas 2D + Vite
- **语言**：TypeScript
- **复杂度**：简单（1 session）
- **3 句话场景**：「你打开 CC，说'做个贪吃蛇'。CC 10 秒生成了代码——能跑。但你想加排行榜、调速度、改 emoji 主题。你又说了一遍。CC 生成了完全不同的代码。你意识到：告诉 CC **怎么做**不是靠说——是靠写。」
- **是什么**：用 CC 从零做出一个贪吃蛇网页游戏。学会写第一个 Skill（`/add-feature` 模板），学会用 Skill 让 CC 按固定模式加新功能。
- **为什么**：这是读者的第一个作品。1 小时内做出一个能发给朋友的链接——这是"我可以用 CC 做东西"的证明。Skill 第一次出现——不是讲 Skill 的理论，是让读者自己感受到"没 Skill 时多痛苦、有 Skill 时多爽"。
- **CC 功能聚焦**：**Skills（初体验）**——写 `/add-feature snake-xxx` skill 模板
- **自检（读者必须自己做）**：让贪吃蛇每秒加速 5%。做完才能读下一章。

### 04. 贪吃蛇 + 全球排行榜（下）：Memory + Hooks——你的游戏活了

- **类型**：项目章（小游戏续）
- **标签**：🎮 游戏
- **项目**：Snake + 排行榜（续）— Supabase + Hooks + Memory
- **语言**：TypeScript
- **复杂度**：简单（1 session）
- **3 句话场景**：「凌晨两点，你的 Discord 响了。朋友说：'你的贪吃蛇排行榜被人刷了——有人用脚本提交了 99999 分。'你打开排行榜数据库——发现分数表里全是垃圾数据。游戏做出来 ≠ 游戏能活。」
- **是什么**：给贪吃蛇加上全球排行榜（Supabase），配置 CLAUDE.md 让 CC 记住项目规范（emoji 蛇头、速度单位、分数公式），装上第一个 Hook（每次改代码自动跑 `tsc --noEmit`）。
- **为什么**：排行榜让"自嗨"变"跟人比"——这是情绪升级。Memory 的初体验不是讲 CLAUDE.md 的 4 级加载链——是让读者自己撞上"每次都要重讲规则"的痛苦。Hooks 第一次出现——不是安全大道理，是"你改了代码 CC 没检查类型，排行榜挂了"。
- **CC 功能聚焦**：**Memory（初体验）** + **Hooks（初体验）**——PostToolUse 自动类型校验。两个功能共享同一个故事——"排行榜被刷"的危机自然地引出"记住规范"和"拦截危险操作"两个需求。
- **自检**：加一个 Hook 拦截所有 `DROP TABLE` 语句。5 分钟。做到？下一章。

## 篇二：核心机制深度（5 章）

> 情绪锚点：**从"能用"到"精通"**。读者已经做出了第一个作品，现在回头深入理解每个功能——每个功能一篇聚焦文章，从"你在做项目时撞上的真实痛点"进入。

### 05. Hooks 深度：凌晨两点，CC 在你睡觉时"优化"了数据库

- **类型**：功能章（深度）
- **标签**：🎮 游戏（延续贪吃蛇场景）
- **项目场景**：贪吃蛇游戏的排行榜数据库被 CC 的"自动优化"删了
- **语言**：TypeScript + Bash
- **复杂度**：中等
- **3 句话场景**：「你的贪吃蛇上线一周了。凌晨两点，排行榜挂了。你查日志——CC 在执行你白天给的一个模糊指令时，'顺便'清理了 `/var/lib`。你没有 Hook。你甚至不知道 Hook 是什么。第二天早上，你写了第一个 PreToolUse hook。」
- **是什么**：5 类 handler × 3 层事件模型的完整讲解。从"拦截 rm -rf"的安全底线开始，到"自动跑测试 → 自动修 lint → 自动 commit message 校验"的完整工作流。白名单策略 vs 黑名单策略的 trade-off。PreToolUse vs PostToolUse 的选用决策树。
- **为什么（为什么不在第 3 章就讲这个？）**：因为读者还没撞过墙。Hooks 是安全带——但没人会在没坐过车的时候对安全带感兴趣。读者先做项目（03-04），撞上"CC 可以干任何事"的恐惧，然后 Hooks 章不是"你应该学"——是"你现在就需要"。
- **CC 功能聚焦**：**Hooks 深度**
- **认知失调触发器**：`npm run clean` 调了 `rm -rf ./dist`。npm 在白名单里，rm 在黑名单里。到底拦还是不拦？
- **留白**：读者改造自己的 pre-bash-validator.sh，补上 `rm -rf ~` 的漏洞。不补 = 第 06 章开头看不懂。

### 06. Memory 系统：你每次开新 session 都要重讲一遍项目规范

- **类型**：功能章（深度）
- **标签**：多个项目场景（贪吃蛇、Rust CLI、个人博客）
- **项目场景**：读者维护 3 个项目，每次切项目开新 session，CC 都像第一次见面
- **语言**：TypeScript / Rust / Markdown
- **复杂度**：中等
- **3 句话场景**：「你同时维护三个项目：贪吃蛇（TypeScript）、博客（Next.js）、一个 Rust CLI。每次打开新 session，CC 都像第一次见你——它不知道你用 React 还是 Vue，不知道你讨厌分号还是喜欢分号。你花了 20 分钟重新解释。第三次的时候你受不了了。」
- **是什么**：4 级 CLAUDE.md 加载链（user → project → session → local）、auto memory 自动笔记、Memory 文件的 frontmatter 约定。`/memory` 命令实战。不同项目类型的 CLAUDE.md 最佳实践（游戏项目记"蛇头用 emoji、速度单位 ms"、Rust 项目记"macroquad API、借用检查器常见错误"、商业项目记"品牌调性、订阅者偏好"）。
- **CC 功能聚焦**：**Memory 深度**
- **认知失调触发器**：「你觉得你的 CLAUDE.md 写得很好了。现在关掉这个 session，新开一个——问 CC '我们的项目用什么语言？'它能答对吗？」
- **留白**：给 Rust CLI 项目写一个 CLAUDE.md。下一章的 Skill 实战会用到这个项目。

### 07. Skill 实战：从需求到发布，写一个 ≥12 分的 Skill

- **类型**：功能章（深度）
- **标签**：多个项目场景
- **项目场景**：读者已经在 03-04 写过简单的 Skill，现在需要学会系统地设计和评估
- **语言**：Markdown（Skill 文件）
- **复杂度**：中等
- **3 句话场景**：「你上次写的 `/add-feature` skill 用了两周。今天你想加一个新功能——CC 又生成了完全不同的代码。你打开那个 skill 文件——发现自己写得太烂了。'我明明写了规则，为什么它不听话？'因为写 Skill 不是写规则——是写给 CC 的剧本。」
- **是什么**：8 建议 0/1/2 评分系统（每项 0/1/2，满分 16，≥12 分可用）。Skill 文件结构：name + description（触发条件）+ 正文（SOP + 约束 + 示例）。需求识别：什么时候该写 Skill，什么时候该用 CLAUDE.md？实战：从一个模糊需求（"想让 CC 自动做部署"）出发，经历需求澄清 → 边界定义 → 初版 → 跑分 → 改版 → 终版，完整走一遍。
- **CC 功能聚焦**：**Skill 实战**
- **为什么**：反复教 CC 同一件事是时间黑洞。Skill 的本质是把"口述经验"变成"可复用资产"。写得不好比没写更糟（占 context 但帮不上忙）。
- **自检**：用 8 建议 0/1/2 评分表给自己写的 Skill 打分。不到 12 分？改。

### 08. MCP：你的游戏需要查数据库排行榜。CC 说"我访问不了"

- **类型**：功能章（深度）
- **标签**：🎮 游戏 + 💰 商业 + 🌐 网站
- **项目场景**：读者想给贪吃蛇加一个"今日最高分"功能——CC 需要直接查 Supabase 数据库
- **语言**：TypeScript + JSON（MCP 配置）
- **复杂度**：中等
- **3 句话场景**：「你想给贪吃蛇加一个功能——玩家点'今日最高分'，CC 直接查 Supabase 数据库返回结果。你说：'帮我查一下 scores 表今天的最高分。'CC 说：'我没有数据库访问权限。'你愣了一下：'你不是 Claude Code 吗？''我是。但我没有数据库的钥匙。'」
- **是什么**：3 种 transport（stdio / SSE / streamable HTTP）的选型逻辑。配好一个 Supabase MCP server + 理解 MCP 的"工具 → 资源 → 提示"三层模型。MCP 不是"额外功能"——没有 MCP，CC 就是个聪明的孤岛。实战：给贪吃蛇项目接上 Supabase MCP，CC 能直接查排行榜、看用户数据。
- **CC 功能聚焦**：**MCP**
- **认知失调触发器**：「你配好了 Supabase MCP。现在试试让 CC 同时查本地 SQLite 和远程 Supabase——两个 MCP server 能一起用吗？」
- **留白**：给你的 Markdown→微信转换器项目配一个 MCP server。下一章讲 Sub-agents 时会用到。

### 09. Sub-agents：物理调参 + 关卡设计——两个 agent 同时干活

- **类型**：功能章（深度）
- **标签**：🎮 游戏（Phaser 平台跳跃）
- **项目场景**：Phaser 2D 平台跳跃项目——物理手感调参和关卡设计是两类完全不同的工作，一个人做太慢
- **语言**：TypeScript
- **复杂度**：中等
- **3 句话场景**：「你的 2D 平台跳跃游戏卡住了。跳跃手感不对——重力、速度、碰撞，调一个参数改一个，一晚上过去了。而关卡设计还一动没动。你说：'CC，能不能物理调参和关卡设计同时做？'CC 说：'能。但不是我一个人做。'」
- **是什么**：什么时候用 sub-agent（并发任务 × 独立 context × 不同专业方向）。Worktree 隔离——两个 agent 改不同文件，互不踩踏。Agent 类型选择（general-purpose / Explore / Plan）。并行 vs Pipeline 的决策树。实战：同时起两个 agent——一个调 Phaser 物理参数（重力 9.8→12？速度 300→450？），一个设计 3 个新关卡（生成 .json 关卡文件）。
- **CC 功能聚焦**：**Sub-agents**
- **为什么**：单个 context 装不下复杂任务。Sub-agents 的本质不是分工——是隔离。用错了要么浪费 token，要么互相踩踏。
- **留白**：给你的 Rust CLI 项目写一个 sub-agent 架构——哪些活能并行？哪些必须串行？

## 篇三：桌面应用实战 — Tauri 全家桶（3 章）

> 情绪锚点：**从网页到桌面，专业感翻倍**。父项目 Maestro 本身是 Tauri 栈——读者做这些项目等于实战 Tauri 2（桌面配置 + Rust 后端 + IPC + 打包分发），是本教程核心差异化。

### 10. Tauri 入门：贪吃蛇从 Web 到桌面——双击图标就开

- **类型**：项目章（桌面应用迁移）
- **标签**：🎮 游戏 + 🖥️ 桌面
- **项目**：Tauri 桌面贪吃蛇 — TypeScript + Rust + Tauri 2 + SQLite
- **语言**：TypeScript + Rust
- **复杂度**：中等（2–3 session）
- **3 句话场景**：「你的网页贪吃蛇做得不错。但每次打开都要先 `npm run dev`，浏览器地址栏占了屏幕 1/6。朋友说：'你能给我一个双击就开的版本吗？像 VS Code 那样。'你说：'能。但需要学一个叫 Tauri 的东西。'」
- **是什么**：Tauri 2 项目从零搭建。理解 Tauri 架构——Rust 后端（commands、SQLite、系统托盘）+ WebView 前端（Canvas 贪吃蛇逻辑复用）。核心概念：`#[tauri::command]` / `invoke()` 前后端通信 / `tauri.conf.json` capabilities 权限矩阵 / 跨平台窗口管理。实战：把 Web 版贪吃蛇迁移到 Tauri——排行榜从 Supabase 换成 SQLite 本地存储、系统通知（破纪录时 OS notification）、全局快捷键唤起、应用菜单。
- **为什么**：这是本教程的核心差异化章节。Tauri 2 是父项目 Maestro 的技术栈——读者学完之后可以直接扩展父项目。从 Web 到桌面的迁移不是"换技术栈"——是"升级体验"。**严格按"先 Web 版（03-04）→ 再桌面版（本章）"的顺序**——读者先理解"做什么"，再理解"怎么包装成桌面应用"。
- **CC 功能聚焦**：**Tauri 2 全栈**——Skills（`/add-frontend-feature` + `/add-rust-command` 两个模板）、Memory（CLAUDE.md 记 Tauri 2 关键约定）、Hooks（保存 .rs 跑 `cargo check`，保存 .ts 跑 `tsc --noEmit`）、Sub-agents（Rust 后端 agent + 前端 Canvas agent 并行）
- **最大坑（前置告知）**：Tauri 2 首次配置（`tauri.conf.json` / capabilities 权限矩阵 / 资源加载路径）会踩多次。macOS codesign + notarize。WebView 跨平台一致性差异。

### 11. Tauri Markdown 笔记：做一个完全本地的 Obsidian 替代品

- **类型**：项目章（桌面工具）
- **标签**：🌐 工具 + 🖥️ 桌面
- **项目**：Tauri Markdown 笔记 — Rust + TypeScript + SQLite FTS5 + CodeMirror 6 + Marked.js
- **语言**：Rust + TypeScript
- **复杂度**：中等（2–3 session）
- **3 句话场景**：「你打开 Notion——它在加载。你打开 Obsidian——它说你的 vault 太大，建议付费 sync。你只是想写个 Markdown 笔记。你的数据凭什么存在别人的服务器上？你打开 CC，说：'帮我做一个。存在我的硬盘上。'」
- **是什么**：文件系统监听（`notify` crate）扫描本地 .md 文件夹 + 双链解析（`[[文件名]]` 自动识别 + 反向链接）+ 标签系统 + 全文搜索（SQLite FTS5 + jieba-rs 中文分词）+ 三栏布局（文件树 + CodeMirror 6 编辑 + Marked.js 预览）。实战重点：FTS5 中文分词（默认对中文不友好——不处理 = 中文搜索几乎不可用，这是真实的工程坑）。
- **为什么**：这是"做到一半你会发现真的能用"的项目。不像贪吃蛇是好玩——Markdown 笔记是你每天都会打开的工具。Tauri + 文件系统监听 + SQLite FTS5 是桌面工具开发的"黄金组合"——学完这章，读者可以自己做任何本地优先的桌面工具。

### 16. 工程方法论：让 CC 按流程干活

- **类型**：方法论章
- **标签**：所有项目场景
- **项目场景**：读者做了一个新功能直接上线——挂了。没有测试、没有 review、没有规划。
- **语言**：TypeScript / Rust
- **复杂度**：中等
- **3 句话场景**：「你做的新功能上线了。10 分钟后，用户开始报 bug。你让 CC 修——它又引入了一个新 bug。你想起一句话：'CC 写代码很快。但如果没测试、没 review、没规划——'快'就是'灾难加速'。'你需要的不是更聪明的 CC——是让 CC 走流程。」
- **是什么**：brainstorming → worktree → plans → subagent → TDD → review → finish 7 步工作流。每一步的"什么时候用、什么时候跳过"决策树。实战：拿前面做过的 Markdown→微信转换器，用完整工作流重做一次——让读者体验"有流程的 CC"和"裸奔的 CC"的区别。
- **CC 功能聚焦**：**7 步工作流**
- **为什么**：自由发挥的 CC = 快但不可靠；按流程走的 CC = 快且可预期。方法论不是"额外负担"——是用流程把 CC 的速度约束在质量轨道上。
- **认知失调触发器**：「你觉得你的代码没问题。现在不让 CC 跑测试——直接 commit、push、上线。你敢吗？」

### 17. Git 工作流：worktree 隔离 + Conventional Commits + PR 合入

- **类型**：方法论章
- **标签**：所有项目场景
- **项目场景**：读者同时改 3 个功能，commit 时全乱了
- **语言**：Git / TypeScript / Rust
- **复杂度**：中等
- **3 句话场景**：「你同时改了三个功能——贪吃蛇加速、排行榜翻页、主题切换。commit 的时候——全乱了。三个功能混在一个 commit 里，message 写的是'update'。一周后你回来看——你不知道哪个改了什么。你的 Git 历史是一本谁也看不懂的日记。」
- **是什么**：Worktree 隔离——并行开发互不踩踏（一个 worktree 一个功能，CC 在里面独立工作）。Conventional Commits（feat/fix/chore 规范——让 commit message 不再是"update"和"fix bug"）。PR 合入流程——squash merge vs rebase merge 的场景选择。实战：拿贪吃蛇项目，同时开 3 个 worktree 并行开发 3 个功能，规范 commit，合入 PR。
- **CC 功能聚焦**：**Git 工作流 + Worktree 隔离**
- **为什么**：CC 改代码改完就完了——但代码必须进仓库。Worktree 隔离是 CC 并行工作的基础。Git 是代码的时间轴——CC 可以帮你写，但不能帮你管理历史。
- **自检**：开两个 worktree，同时改贪吃蛇的两个功能，互不影响。做到？下一章。

### 18. Prompt 工程 + 项目配置：跟 CC 说话的艺术 × 让 CC 守规矩的开关

- **类型**：综合章
- **标签**：所有项目场景
- **项目场景**：Prompt 写得太模糊，CC 产出货不对板。settings.json 从来没改过——不知道自己浪费了多少 token 或暴露了多少风险。
- **语言**：TypeScript / Rust / JSON
- **复杂度**：中等
- **3 句话场景**：「你说：'帮我写个网站。'CC 从头建了一个 Next.js 项目——App Router、Tailwind、shadcn/ui 全配好了。你只想要一个 HTML 文件。你看着满屏的配置文件，不知道是夸它聪明还是骂它浪费。你说得越少，它猜得越多。猜得越多，错得越远。」
- **是什么**：Prompt 工程三原则——精确 > 含糊、上下文 > 裸问、分步 > 一口气。反模式案例集（"帮我写个网站" / "修一下这个 bug" / "优化性能"——为什么这些 prompt 是灾难）。项目配置——从头配好 `.claude/settings.json`（权限、模型、token 预算、环境变量）。多模型选择：不是"哪个最聪明选哪个"，是"按任务性价比选"。
- **CC 功能聚焦**：**Prompt 工程** + **Settings 配置**
- **为什么**：同样的工具，不同的人用天壤之别。差距不在模型——在 prompt 和配置。模型的能力是常量，prompt 的质量和 settings 的合理性是变量。提升这两者 = 不花钱的模型升级。
- **自检**：把你的 `.claude/settings.json` 贴给 CC，让它审一遍——有没有安全漏洞？有没有浪费 token 的配置？

- **语言**：TypeScript + Python
- **复杂度**：进阶
- **3 句话场景**：「你找遍了 MCP Registry——没有一个 server 能接你公司的内部 API。你说：'那我写一个。'你说的那一刻——你从 CC 用户变成了 CC 生态的贡献者。你的 server 可以被所有 CC 用户用。你不再是'学 CC 的人'——你是'让 CC 更强的人'。」
- **是什么**：写一个 MCP server（TS + Python 双栈）→ 发布到 npm / PyPI + MCP Registry。理解 MCP 协议的"工具 → 资源 → 提示"三层模型，把它映射到真实 API（公司的内部 API / 私有的数据库 / 定制的工具）。实战：写一个接公司内部 API 的 MCP server，CC 能直接查内部数据。
- **CC 功能聚焦**：**MCP 生态深度**
- **为什么**：市面上 MCP server 覆盖不了所有内部系统。会消费 MCP = 用户；会写 MCP = 生态参与者。生态参与者的价值是指数级的——一个 server 可以被所有 CC 用户用。

### 23. Agent SDK & API 集成：把 CC 嵌入你的系统

- **类型**：知识章
- **标签**：🌐 生态
- **项目场景**：你想让 CI/CD 自动修 bug——不需要人打开 CC，SDK 直接调
- **语言**：TypeScript + Python
- **复杂度**：进阶
- **3 句话场景**：「你团队的 CI 每次挂了，你都要手动打开 CC，复制错误日志，说'修一下'，再把生成的代码贴回去。有一次你休假——CI 挂了三天没人管。你回来那天说：'不能再靠我手动调 CC 了。让系统自己调。'」
- **是什么**：Agent SDK vs CLI 定位差异 + 3 种集成模式（CLI 调用 / SDK 嵌入 / API 直调）。PR review bot——用 SDK 写一个自动 review PR 的 bot。CI/CD 自动修 bug——测试挂了自动调 SDK 修。批量任务——用 API 直调，人不在循环里。
- **CC 功能聚焦**：**Agent SDK + API**
- **为什么**：CLI 是人机对话，SDK 是系统间对话。PR review bot、CI/CD 自动修 bug——这些场景"人"是中间环节，应该去掉。SDK 的本质：把人的判断从循环中移除。
