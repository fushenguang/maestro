## 第一篇：入口

### 00. 教程地图：怎么读这教程

- **是什么**：给读者一个 5 分钟定位系统——4 条阅读路径分流，决定从哪章开始。
- **为什么（第一性原理）**：教程不是小说，没人从头读到尾。读者带着具体问题来，需要最快路径找到答案。如果前 5 分钟没让他觉得"这教程能解决我的问题"，他就走了。路径分流的本质是降低决策成本——不是让读者看完所有章，是让他只看该看的章。

## 第二篇：上手 — 从零到第一个项目

### 01. Claude Code 是什么 & 环境配置

- **是什么**：装好 CC + 跑通第一个 prompt。
- **为什么（第一性原理）**：如果装不上，后面所有内容都作废。中国市场有独特障碍——VPN、支付、封号——这些不是"附加信息"，是"能不能用的前提"。先解决"能不能用"，再谈"怎么用好"。

### 02. 对话基础：slash commands & 权限系统

- **是什么**：掌握常用 slash commands + 理解权限弹窗的三种选择（allow / deny / always allow）什么时候用。
- **为什么（第一性原理）**：CC 是对话驱动工具，不是 IDE 插件。用户需要知道"能说什么"和"CC 能做什么"。权限系统是安全感的来源——不知道什么时候该点 allow，用户要么全点 yes（不安全），要么全点 no（用不了）。两者都导致弃用。

## 第三篇：核心机制 — 让 CC 听话、记住、复用

### 03. Hooks：让 CC 守规矩的 8 条铁律

- **是什么**：写一个 PreToolUse hook 拦截危险命令——rm -rf /、sudo、secret 读取，全部挡住。
- **为什么（第一性原理）**：CC 以用户全权限跑——这是所有安全问题的根。hook 是唯一能在 CC 动手之前拦截的机制。不是"锦上添花"，是"不犯致命错的底线"。没有 hook 的 CC 像一个有 root 权限但没 code review 的实习生。

### 04. 记忆系统：跨会话记得你

- **是什么**：配好 4 级 CLAUDE.md 加载链 + auto memory 自动笔记。
- **为什么（第一性原理）**：每次新 session CC 都像第一次见面——重复教同样的事是所有 CC 用户的第一痛点。记忆不是"让 CC 更聪明"，是"不让它每次都变笨"。跨会话知识 = 效率的复利。

### 05. Skill 哲学：怎么把经验写给 CC 用

- **是什么**：理解 Skill 是什么 + 能用 8 建议 0/1/2 评分表判断一个 Skill 写得好不好。
- **为什么（第一性原理）**：反复教 CC 同一件事是时间黑洞。Skill 的本质是把"口述经验"变成"可复用资产"。但 Skill 有设计方法论——写得不好比没写更糟（占 context 但帮不上忙）。先教会判断标准，再教怎么写。

### 06. Skill 实战：从零写一个可复用 Skill

- **是什么**：写出一个评分 ≥ 12 的 Skill 文件——从需求识别到发布完整走一遍。
- **为什么（第一性原理）**：05 章给了理论，但理论和能写之间有一条鸿沟——读者需要看到完整的"从需求→成品"过程。不是 showcase，是 walkthrough。只讲原则不讲步骤 = 听懂了但不会做。

### 07. MCP：让 CC 接上外部世界

- **是什么**：配好一个 MCP server + 理解 3 种 transport（stdio / SSE / streamable HTTP）的选型逻辑。
- **为什么（第一性原理）**：CC 只能读本地文件。但真实项目需要查数据库、调 API、搜文档。MCP 是 CC 的"外部感官"——没有它，CC 就是个聪明的孤岛。MCP 协议是 CC 生态的脊柱——server 越多，CC 越强。

### 08. Sub-agents：把活派出去

- **是什么**：理解什么时候用 sub-agent + 会配 memory 字段做 worktree 隔离。
- **为什么（第一性原理）**：单个 context 装不下复杂任务。但拆解委派不是"多开几个 CC"——sub-agent 有独立 context、独立 memory、worktree 隔离。用错了要么浪费 token，要么互相踩踏。委派的本质不是分工，是隔离。

## 第四篇：工程实践 — 用 CC 做正经项目

### 09. 工程方法论：让 CC 按流程干活

- **是什么**：理解 brainstorming → worktree → plans → subagent → TDD → review → finish 7 步工作流。
- **为什么（第一性原理）**：CC 写代码很快——但如果没测试、没 review、没规划，"快"就是"灾难加速"。方法论不是"额外负担"，是用流程把 CC 的速度约束在质量轨道上。自由发挥的 CC = 快但不可靠；按流程走的 CC = 快且可预期。

### 10. Git 工作流：worktrees、commit、PR

- **是什么**：掌握 CC 下的 Git 最佳实践——worktree 隔离、conventional commits、PR 合入。
- **为什么（第一性原理）**：CC 改代码改完就完了——但代码必须进仓库。worktree 隔离是 CC 并行工作的基础。不懂 worktree = 要么冲突，要么丢代码。Git 是代码的时间轴——CC 可以帮你写，但不能帮你管理历史。

### 11. Prompt 工程：跟 CC 说话的艺术

- **是什么**：掌握给 CC 写 prompt 的核心技巧——精确 > 含糊、上下文 > 裸问、分步 > 一口气。
- **为什么（第一性原理）**：同样的工具，不同的人用天壤之别。差距不在模型——在 prompt。模型的能力是常量，prompt 的质量是变量。提升 prompt 质量 = 不花钱的模型升级。

### 12. 项目配置全指南：settings.json & 多模型

- **是什么**：能从头配好一个生产级 `.claude/settings.json`——权限、模型、token 预算、环境变量。
- **为什么（第一性原理）**：默认配置是"能用"——生产级需要"好用且安全"。多模型选择不是"哪个最聪明选哪个"，是"按任务性价比选"。Token 预算不是调参，是成本控制。配置 = 对 CC 行为的编程。

## 第五篇：进阶 — 团队 & 生态

### 13. 企业部署 & 治理

- **是什么**：理解 managed policy + 审计日志 + 合规（SOC2 / ISO27001）三件套。
- **为什么（第一性原理）**：个人用 CC = 自己的风险自己担。团队用 CC = 一个人的错误全团队买单。企业治理不是"限制自由"，是让 CC 在组织风险边界内运行。0 信任模型的本质：不信任任何人——包括写好 hook 的你自己。

### 14. 同类工具横向对比

- **是什么**：能根据场景从 CC / Cursor / Codex / Aider / Gemini CLI 中选对工具。
- **为什么（第一性原理）**：没有一个工具在所有场景最优。选错工具比不用工具更糟——投入了学习成本但产出不匹配。决策矩阵 > 推荐清单——因为"适合"取决于场景，不是取决于工具。

### 15. MCP 生态深度：server 开发 & 发布

- **是什么**：能写一个 MCP server（TS + Python 双栈）并发布到 npm / PyPI + Registry。
- **为什么（第一性原理）**：市面上 MCP server 覆盖不了所有内部系统。公司的 API、私有的数据库、定制的工具——必须自己写。会消费 MCP = 用户；会写 MCP = 生态参与者。生态参与者的价值是指数级的——一个 server 可以被所有 CC 用户用。

### 16. Agent SDK & API 集成

- **是什么**：理解 SDK vs CLI 定位差异 + 3 种集成模式（CLI 调用 / SDK 嵌入 / API 直调）。
- **为什么（第一性原理）**：CLI 是人机对话，SDK 是系统间对话。PR review bot、CI/CD 自动修 bug——这些场景"人"是中间环节，应该去掉。SDK 的本质：把人的判断从循环中移除。

## 第六篇：实战案例 & 附录

### 17. Cookbook：真实场景实战

- **是什么**：跟着做 3-5 个完整案例——从 0 建全栈应用、重构遗留代码、自动化 CI/CD。
- **为什么（第一性原理）**：看完所有理论 ≠ 会用。学习编程不是看书学会的——是跟着敲代码学会的。CC 教程同理——需要一个从头到尾"看着做"的过程。Cookbook 的本质是消除"知道但不会做"的鸿沟。

### 18. 常见问题排查 & 踩坑集

- **是什么**：遇到问题 30 秒定位到答案——按症状索引，不按章节索引。
- **为什么（第一性原理）**：出问题的时候读者没耐心从头看教程。排查指南的价值 = 定位速度 × 准确度。按症状索引（不是按章节索引）是一性原理——用户知道"发生了什么"，不知道"该去哪章找"。从症状出发 = 0 跳转成本。

---

### 附录 A. Slash Commands 速查表

- **是什么**：全部 slash commands 按类别排列 + 一句话 + 示例。
- **为什么（第一性原理）**：命令不需要"学"，需要"查"。速查表定位时间是 0 秒（打开就有），教程定位时间是 N 秒（需要回忆在哪个章节）。速查表 = 降低重复查找的摩擦力。

### 附录 B. 资源索引

- **是什么**：官方文档 / 社区资源 / 推荐 Skill 库 / MCP server 列表。
- **为什么（第一性原理）**：教程不是知识的终点，是入口。读者学完需要知道"下一步去哪"。资源索引是教程的自知之明——知道什么该自己教，什么该指路。好的索引 = 读者不迷路。
